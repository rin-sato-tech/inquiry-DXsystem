# 発展課題

## 1. 目的

本ドキュメントは、社内問い合わせ管理システムの今後の発展課題を整理するものです。

本システムは、Python、SQLite、Streamlit、pandas、Tableauを用いて作成した簡易DXアプリです。
Ver.1では、問い合わせの受付、記録、進捗管理、期限超過判定、集計、Tableau連携を対象としました。Ver.2では、Ver.1の問い合わせ管理機能を拡張し、以下の機能を追加しています。

- 要対応アラート
- FAQ候補管理
- カテゴリ別入力フォーム
- 依頼者向け確認画面
- Ver.2追加集計
- Tableau出力列の拡張

一方で、本格運用を想定すると、認証、権限管理、通知、FAQ公開、操作履歴、DB移行など、さらに検討すべき課題があります。

## 2. 現在の位置づけ

本システムは、社内問い合わせ管理業務を題材にしたポートフォリオ用の簡易DXアプリです。対象企業は、従業員30名規模の架空企業「株式会社西日本アーバン設備」を想定しています。
現在のシステムは、実運用そのものではなく、以下を示すことを目的としています。

- 業務課題の整理
- 要件定義
- データ設計
- SQLiteによるデータ管理
- Streamlitによる業務画面作成
- pandasによる集計処理
- Tableau連携用CSV出力
- テスト・保守手順の整理
- 段階的な機能拡張

そのため、本格的な社内システムとして利用するには、今後さらに機能面・技術面・運用面の拡張が必要です。

## 3. Ver.2で対応したこと

Ver.2では、Ver.1の「問い合わせを記録・管理する仕組み」から一歩進めて、問い合わせ削減や自己解決支援につながる機能を追加しました。

| 項目                   | 内容                                                             |
| ---------------------- | ---------------------------------------------------------------- |
| 要対応アラート         | 期限超過、本日期限、期限間近、担当者未設定、情報待ち長期化を表示 |
| FAQ候補管理            | 完了済み問い合わせからFAQ候補を登録・更新・解除                  |
| カテゴリ別入力フォーム | 問い合わせカテゴリに応じた追加情報を登録                         |
| 依頼者向け確認画面     | 問い合わせIDまたは依頼者名で問い合わせ状況を確認                 |
| Ver.2追加集計          | アラート、FAQ候補、追加情報入力率、依頼者向け表示件数などを集計  |
| Tableau出力拡張        | Ver.2追加列をTableau用CSVに出力                                  |
| Ver.2スモークテスト    | Ver.2追加機能の基本動作を確認するテストを追加                    |
| DB移行処理             | 既存DBにVer.2追加カラムを追加する処理を作成                      |

## 4. Ver.2で一部対応したが、今後も拡張余地があるもの

Ver.2で実装した機能の中にも、本格運用に向けては追加検討が必要なものがあります。

| 項目                   | Ver.2での対応                                    | 今後の課題                                     |
| ---------------------- | ------------------------------------------------ | ---------------------------------------------- |
| FAQ候補管理            | FAQ候補の登録・更新・解除まで実装                | FAQ公開ページ、FAQ検索、公開可否管理が必要     |
| 依頼者向け確認         | 問い合わせID・依頼者名で検索できるデモ画面を実装 | ログイン認証、本人確認、権限管理が必要         |
| 要対応アラート         | アプリ画面上でアラートを表示                     | メール・Slack通知、期限前リマインドが必要      |
| カテゴリ別入力フォーム | カテゴリに応じた追加情報を入力可能               | 項目の必須制御、カテゴリ別バリデーションが必要 |
| Tableau出力            | Ver.2追加列をCSV出力                             | ダッシュボード側の設計更新が必要               |

## 5. 機能面の発展課題

### 5.1 認証・権限管理

| 課題         | 内容                                                       |
| ------------ | ---------------------------------------------------------- |
| ログイン認証 | 利用者を識別し、本人の問い合わせだけを表示できるようにする |
| 権限管理     | 依頼者、管理部、管理者で閲覧・更新範囲を分ける             |
| ロール管理   | 依頼者、担当者、管理者などの役割を定義する                 |
| 表示制御     | 管理部内部向け項目を依頼者に表示しないよう制御する         |

現在の依頼者向け確認画面はデモ用です。本格運用では、ログイン認証を行い、依頼者本人の問い合わせだけを表示する必要があります。

### 5.2 通知機能

| 課題               | 内容                                             |
| ------------------ | ------------------------------------------------ |
| 期限前通知         | 希望期限が近い問い合わせを担当者に通知する       |
| 期限超過通知       | 期限超過した問い合わせを担当者・管理者に通知する |
| ステータス変更通知 | ステータスが更新された際に依頼者へ通知する       |
| FAQ候補通知        | FAQ候補に登録された問い合わせを管理者へ通知する  |
| メール連携         | 登録・更新・期限通知をメールで送信する           |
| Slack連携          | 社内チャットにアラートや更新情報を通知する       |

Ver.2では画面上に要対応アラートを表示していますが、アプリを開かないと気づけません。実運用では、通知機能を追加することで対応漏れをさらに減らせます。

### 5.3 FAQ機能の拡張

| 課題            | 内容                                          |
| --------------- | --------------------------------------------- |
| FAQ公開ページ   | FAQ候補を依頼者向けに公開する                 |
| FAQ検索         | 問い合わせ前にFAQを検索できるようにする       |
| FAQ公開可否     | FAQ候補を公開するか、非公開にするかを管理する |
| FAQカテゴリ管理 | FAQをカテゴリごとに整理する                   |
| FAQ編集履歴     | FAQタイトルや回答案の編集履歴を残す           |
| FAQ利用状況分析 | どのFAQがよく閲覧されているか分析する         |

Ver.2ではFAQ候補を蓄積する段階まで実装しています。今後は、FAQを依頼者向けに公開し、問い合わせ前に自己解決できる仕組みへ発展させることが考えられます。

### 5.4 対応履歴・コメント管理

| 課題           | 内容                               |
| -------------- | ---------------------------------- |
| コメント履歴   | 対応過程のやり取りを履歴として残す |
| ステータス履歴 | ステータス変更の履歴を残す         |
| 担当者変更履歴 | 担当者が変更された履歴を残す       |
| 対応メモ       | 管理部内部向けのメモを残す         |
| 操作履歴       | 誰がいつ何を更新したかを記録する   |

現在は、問い合わせ1件に対して最新のステータスや対応内容を保存しています。本格運用では、対応過程を時系列で残すことで、引き継ぎや監査に対応しやすくなります。

### 5.5 添付ファイル対応

| 課題                   | 内容                                               |
| ---------------------- | -------------------------------------------------- |
| スクリーンショット添付 | PC・システム関連のエラー画面を添付できるようにする |
| 請求書添付             | 経費・請求関連の証憑資料を添付できるようにする     |
| 申請書添付             | 契約・書類関連のファイルを添付できるようにする     |
| ファイル管理           | 添付ファイルの保存場所、容量、権限を管理する       |

現在はテキスト情報のみを管理しています。実運用では、エラー画面や証憑資料を添付できると、確認往復をさらに減らせます。

### 5.6 承認フロー

| 課題         | 内容                                         |
| ------------ | -------------------------------------------- |
| 上長承認     | 権限付与や高額備品購入に承認ステップを設ける |
| 管理者承認   | FAQ公開や重要な変更に承認を設ける            |
| 承認待ち一覧 | 承認待ちの問い合わせを一覧で確認する         |
| 承認履歴     | 誰がいつ承認したかを記録する                 |

Ver.2ではステータスとして「承認待ち」は扱えますが、本格的な承認ワークフローは未実装です。承認フローは仕様が複雑化しやすいため、今後の拡張対象とします。

## 6. 分析面の発展課題

| 課題               | 内容                                                   |
| ------------------ | ------------------------------------------------------ |
| FAQ効果分析        | FAQ候補管理により問い合わせ件数が減ったか分析する      |
| カテゴリ別傾向分析 | 問い合わせカテゴリごとの件数・作業時間を分析する       |
| 追加情報入力率分析 | カテゴリ別入力フォームがどの程度使われているか分析する |
| アラート傾向分析   | 期限超過や担当者未設定がどのカテゴリで多いか分析する   |
| 担当者負荷分析     | 担当者ごとの件数・作業時間・アラート件数を分析する     |
| 月次推移分析       | 問い合わせ件数、作業時間、期限超過件数の推移を確認する |
| 対応時間予測       | カテゴリや優先度から対応時間を予測する                 |
| カテゴリ自動分類   | 問い合わせ内容からカテゴリを推定する                   |

Ver.2では、分析に使える列をTableau出力に追加しています。今後は、Tableauダッシュボード側でVer.2追加列を活用し、改善状況を可視化することが課題です。

## 7. 技術面の発展課題

| 課題                       | 内容                                             |
| -------------------------- | ------------------------------------------------ |
| PostgreSQL移行             | 複数人利用や同時更新に強いDBへ移行する           |
| API化                      | 問い合わせ登録・更新処理をAPIとして分離する      |
| Flask / Django / FastAPI化 | Streamlit以外のWebアプリ構成へ発展させる         |
| フロントエンド分離         | Reactなどを用いて画面とAPIを分離する             |
| Docker化                   | 環境構築を簡単にする                             |
| クラウド配置               | 社内利用を想定してクラウド上にデプロイする       |
| テスト拡充                 | pytestなどを使い、より細かい単体テストを追加する |
| CI導入                     | GitHub Actionsなどでテストを自動実行する         |

現在は、ポートフォリオとして再現しやすいように、SQLiteとStreamlitを採用しています。本格運用に近づける場合は、DB、アプリ構成、テスト、デプロイの見直しが必要です。

| 課題                   | 内容                                         |
| ---------------------- | -------------------------------------------- |
| ユーザーテーブル       | 依頼者、管理部担当者、管理者を管理する       |
| 権限管理テーブル       | ロールごとの閲覧・更新権限を管理する         |
| FAQ専用テーブル        | 問い合わせ履歴から独立してFAQを管理する      |

Ver.2では、実装をシンプルに保つため、基本的に `inquiries` テーブルを中心に管理しています。ただし、本格運用ではテーブルを分割し、問い合わせ履歴、FAQ、ユーザー、権限、添付ファイルなどを分けて管理する方が適しています。

| 入力ルール整備       | 問い合わせカテゴリや優先度の選び方を明確にする       |

## 10. ノーコード・ローコードとの比較

実務上は、以下のような選択肢も考えられる。

- Googleフォーム
- Googleスプレッドシート
- AppSheet
- kintone
- Microsoft Forms
- Power Apps

問い合わせ受付だけであれば、GoogleフォームやMicrosoft Formsでも対応できます。一方で、本プロジェクトでは、問い合わせ受付後の進捗管理、期限超過判定、作業時間集計、要対応アラート、FAQ候補管理、依頼者向け確認、Tableau連携までを対象としました。
そのため、Python、SQLite、Streamlitを用いて、問い合わせ管理業務全体を支援する簡易システムとして実装しています。

## 11. 今後の優先順位

今後拡張する場合は、以下の優先順位が妥当です。

| 優先度 | 課題                      | 理由                                                  |
| ------ | ------------------------- | ----------------------------------------------------- |
| 高     | FAQ公開ページ・FAQ検索    | 問い合わせ件数そのものの削減につながるため            |
| 高     | メール・Slack通知         | 期限超過や対応漏れの予防につながるため                |
| 高     | ログイン認証              | 依頼者向け確認画面を本格運用するために必要            |
| 中     | 権限管理                  | 依頼者、管理部、管理者で表示範囲を分けるため          |
| 中     | 操作履歴                  | 誰がいつ更新したかを確認できるようにするため          |
| 中     | 添付ファイル対応          | PCトラブルや請求書関連の確認往復を減らせるため        |
| 中     | Tableauダッシュボード改善 | Ver.2追加列を使って改善効果を見せやすくするため       |
| 低     | AI自動分類                | データ量が増えてから検討する方がよいため              |
| 低     | 本格承認フロー            | 仕様が複雑化するため段階的に検討する方がよいため      |
| 低     | Webアプリ化               | 現段階ではStreamlitでポートフォリオ目的を満たせるため |

## 12. Ver.3で実装するなら

Ver.3として拡張する場合は、以下のような構成が考えられます。

| 機能          | 内容                                            |
| ------------- | ----------------------------------------------- |
| FAQ公開ページ | FAQ候補を依頼者向けに公開する                   |
| FAQ検索       | カテゴリやキーワードでFAQを検索できるようにする |
| 通知機能      | 期限前・期限超過・ステータス変更時に通知する    |
| ログイン認証  | 依頼者本人の問い合わせだけを表示する            |
| 権限管理      | 依頼者、管理部、管理者で表示・更新範囲を分ける  |
| 操作履歴      | 更新者、更新日時、更新内容を記録する            |
| Tableau更新   | Ver.2追加列を使ったダッシュボードを作成する     |

Ver.3では、Ver.2で作成したFAQ候補管理、要対応アラート、依頼者向け確認画面を、より実運用に近い形へ発展させることが中心になります。

## 13. まとめ

Ver.2では、Ver.1の問い合わせ管理機能を拡張し、要対応アラート、FAQ候補管理、カテゴリ別入力フォーム、依頼者向け確認画面を追加しました。これにより、問い合わせを記録・管理するだけでなく、問い合わせ削減、対応漏れ防止、確認往復の削減、依頼者側の進捗確認まで視野に入れた構成になっています。
今後の発展課題としては、ログイン認証、権限管理、通知機能、FAQ公開ページ、操作履歴、添付ファイル対応、PostgreSQL移行などがあります。
本プロジェクトでは、すべてを一度に実装するのではなく、Ver.1、Ver.2、Ver.3以降に分けて段階的に拡張する方針としています。

## Ver.3で優先的に検討する内容

Ver.3では、Ver.2で蓄積した問い合わせ情報・FAQ候補・アラート情報を、実際の運用に接続することを重視する。

優先的に検討する機能は以下である。

| 優先度 | 項目                   | 内容                                                   |
| ------ | ---------------------- | ------------------------------------------------------ |
| 高     | FAQ公開・検索          | FAQ候補を依頼者が検索できるようにする                  |
| 高     | 簡易ログイン・権限管理 | 依頼者、管理部担当者、管理者で表示範囲を分ける         |
| 高     | 通知対象抽出           | 期限前、期限超過、担当者未設定を通知対象として抽出する |
| 中     | 操作履歴               | 誰がいつ何を更新したかを記録する                       |
| 中     | ステータス履歴         | 問い合わせごとの状態変更履歴を保存する                 |
