# Ver.2 データ定義書

## 1. 目的

本ドキュメントは、社内問い合わせ管理システム Ver.2 で追加・変更したデータ項目を整理するものです。

ルート配下の `docs/data_definition.md` は、Ver.1とVer.2を含めたシステム全体の正式なデータ定義書です。

本ドキュメントでは、Ver.2で追加したカラム、派生列、アラート判定、Tableau出力列を中心に整理します。

## 2. Ver.2データ設計の方針

Ver.2では、Ver.1の `inquiries` テーブルを拡張して、以下の機能に対応します。

- 要対応アラート
- FAQ候補管理
- カテゴリ別入力フォーム
- 依頼者向け確認画面
- Ver.2追加集計
- Tableau出力列の拡張

ただし、Ver.2ではDB構造を複雑にしすぎないため、追加テーブルは作らず、既存の `inquiries` テーブルへのカラム追加を中心に実装します。

本格的なFAQテーブル、ユーザーテーブル、権限管理テーブル、操作履歴テーブルなどはVer.3以降の課題とします。

## 3. 既存テーブル

Ver.1では、問い合わせ情報を `inquiries` テーブルで管理しています。

主な既存カラムは以下です。

| `subcategory`             | サブカテゴリ       |
| `detail`                  | 問い合わせ内容     |

## 4. Ver.2追加カラム

Ver.2では、以下のカラムを追加します。

| カラム名                 | 型      | 初期値 | 内容                                                   |
| ------------------------ | ------- | -----: | ------------------------------------------------------ |
| `faq_candidate`          | INTEGER |      0 | FAQ候補フラグ。1ならFAQ候補、0なら候補ではない         |
| `faq_title`              | TEXT    | 空文字 | FAQ候補として整理した場合のタイトル                    |
| `faq_answer`             | TEXT    | 空文字 | FAQ候補として整理した回答案                            |
| `additional_info`        | TEXT    | 空文字 | カテゴリ別入力フォームで入力された追加情報             |
| `requester_visible`      | INTEGER |      1 | 依頼者向け確認画面に表示するか                         |
| `last_status_changed_at` | TEXT    | 空文字 | 最終ステータス変更日時。将来的な履歴管理・通知機能向け |

## 5. 追加カラムの詳細

### 5.1 `faq_candidate`

FAQ候補かどうかを示します。

|  値 | 意味            |
| --: | --------------- |
|   0 | FAQ候補ではない |
|   1 | FAQ候補である   |

原則として、完了済み問い合わせの中から管理部担当者が選択します。

FAQ候補から外す場合は、`faq_candidate` を0にします。

### 5.2 `faq_title`

FAQとして整理する場合のタイトルを保存します。

例：

- 販売管理システムにログインできない場合の対応
- 共有フォルダの権限追加依頼方法
- 勤怠修正を依頼する場合の必要情報

### 5.3 `faq_answer`

FAQとして整理する場合の回答案を保存します。

問い合わせの `response_summary` をもとに、依頼者向けに分かりやすく書き換えた内容を保存します。

FAQ候補から外しても、Ver.2では `faq_title` と `faq_answer` は下書きとして残します。

### 5.4 `additional_info`

カテゴリ別入力フォームで入力された追加情報を保存します。

Ver.2では、DBカラムをカテゴリごとに増やしすぎないため、追加情報を1つのテキスト項目として保存します。

保存形式は、読みやすさを優先して複数行テキストとします。

例：

```text
PC管理番号: PC-014
発生時刻: 09:30
エラー内容: 販売管理システム起動時に認証エラー
再起動有無: 実施済み
```

### 5.5 `requester_visible`

依頼者向け確認画面に表示するかを示します。

|  値 | 意味                       |
| --: | -------------------------- |
|   0 | 依頼者向け画面に表示しない |
|   1 | 依頼者向け画面に表示する   |

Ver.2では、初期値は1とします。

`requester_visible == 0` の問い合わせは、依頼者向け確認画面には表示しません。

### 5.6 `last_status_changed_at`

ステータスが最後に変更された日時を保存するための項目です。

Ver.2では、将来的なステータス履歴、通知、情報待ち長期化判定への発展を見据えて追加しています。

現時点では、実装上必要に応じて `updated_at` や `request_date` を使って判定を行います。

## 6. カテゴリ別追加情報

カテゴリ別入力フォームでは、以下の追加項目を扱います。

| カテゴリ         | 追加項目                                                 |
| ---------------- | -------------------------------------------------------- |
| PC・システム     | PC管理番号、発生時刻、エラー内容、再起動有無             |
| アカウント・権限 | 対象システム、対象フォルダ・対象機能、必要な権限、承認者 |
| 勤怠・労務       | 対象日、修正前時刻、修正後時刻、理由                     |
| 経費・請求       | 金額、取引先、対象月、添付書類有無                       |
| 備品・設備       | 備品名・設備名、数量、利用目的、希望日                   |

上記に該当しないカテゴリでは、Ver.2では追加項目を表示しない、または自由記述で補足する運用とします。

カテゴリ別入力項目の定義は、`src/category_fields.py` で管理します。

## 7. アラート判定条件

Ver.2のアラート判定条件は以下です。

| アラート種別   | 条件                                                            |
| -------------- | --------------------------------------------------------------- |
| 期限超過       | `status != "完了"` かつ `due_date < 今日`                       |
| 本日期限       | `status != "完了"` かつ `due_date == 今日`                      |
| 期限間近       | `status != "完了"` かつ `今日 < due_date <= 今日 + 1日`         |
| 担当者未設定   | `assignee` が空                                                 |
| 情報待ち長期化 | `status == "情報待ち"` かつ `request_date` から一定日数以上経過 |

情報待ち長期化の日数は、初期値として3日以上とします。

期限間近は、本日期限と重複しないように判定します。

## 9. Tableau出力追加列

Ver.2では、Tableau用CSVに以下の列を追加します。

| 列名                      | 内容                       |
| ------------------------- | -------------------------- |
| `additional_info`         | カテゴリ別追加情報         |
| `has_additional_info`     | 追加情報が入力されているか |
| `has_additional_info_int` | 追加情報ありなら1          |
| `faq_candidate`           | FAQ候補フラグ              |
| `faq_candidate_int`       | FAQ候補なら1               |
| `faq_title`               | FAQタイトル                |
| `faq_answer`              | FAQ回答案                  |
| `requester_visible`       | 依頼者向け表示フラグ       |
| `requester_visible_int`   | 依頼者向け表示なら1        |
| `requester_visible_label` | 依頼者向け表示区分         |
| `alert_overdue`           | 期限超過アラート           |
| `alert_due_today`         | 本日期限アラート           |
| `alert_due_soon`          | 期限間近アラート           |
| `alert_unassigned`        | 担当者未設定アラート       |
| `alert_info_waiting_long` | 情報待ち長期化アラート     |
| `has_alert`               | 何らかのアラート対象か     |
| `alert_type`              | アラート種別               |

Tableauでは、0/1列を合計することで件数を集計できます。

例：

```text
SUM(faq_candidate_int) = FAQ候補件数
SUM(has_additional_info_int) = 追加情報あり件数
SUM(requester_visible_int) = 依頼者向け表示件数
```

また、0/1列の平均を取ることで割合を計算できます。

例：

```text
AVG(has_additional_info_int) = 追加情報入力率
```

## 10. DB移行方針

既存DBにはVer.2追加カラムが存在しないため、移行処理を作成します。

移行処理では、以下を行います。

```text
1. 既存の inquiries テーブル構造を確認する
2. 不足しているVer.2追加カラムを判定する
3. ALTER TABLE で不足カラムを追加する
4. 既存データには初期値を設定する
```

対象ファイルは以下です。

```text
src/migrate_db.py
```

この処理は、既に存在するカラムを再追加しないようにしているため、複数回実行しても問題ありません。

## 11. データ設計上の注意点

Ver.2では、ポートフォリオとして分かりやすい実装を優先します。

そのため、以下の方針とします。

| 方針                         | 内容                                                      |
| ---------------------------- | --------------------------------------------------------- |
| 1テーブル中心                | Ver.2では `inquiries` テーブルを拡張して管理する          |
| 追加情報はテキスト保存       | カテゴリ別追加情報は `additional_info` にまとめて保存する |
| FAQは候補管理まで            | FAQ専用テーブルやFAQ公開ページはVer.3以降とする           |
| 派生列はPython側で計算       | アラート、入力率、表示区分などはDB保存せず計算する        |
| Tableau向け列を用意          | 集計しやすいように0/1列をCSV出力する                      |
| 本格運用との差分を明確にする | ユーザー管理・権限管理・操作履歴は今後の課題とする        |

## 12. 今後のデータ設計上の課題

本格運用を想定する場合、以下の拡張が考えられます。

| 課題                   | 内容                                    |
| ---------------------- | --------------------------------------- |
| ユーザーテーブル       | 依頼者、管理部担当者、管理者を管理する  |
| 権限管理テーブル       | ロールごとの閲覧・更新権限を管理する    |
| FAQ専用テーブル        | 問い合わせ履歴から独立してFAQを管理する |
| FAQ編集履歴テーブル    | FAQ回答案の変更履歴を残す               |
| 操作履歴テーブル       | 誰がいつ問い合わせを更新したか記録する  |
| 添付ファイルテーブル   | スクリーンショットや証憑資料を管理する  |
| ステータス履歴テーブル | ステータス変更の履歴を管理する          |
| 通知履歴テーブル       | いつ誰に通知したかを記録する            |
