# Ver.2 データ定義書

## 1. データ設計の方針

Ver.2では、初期版の `inquiries` テーブルを拡張して、要対応アラート、FAQ候補管理、カテゴリ別入力、依頼者向け確認に対応する。

ただし、Ver.2ではDB構造を複雑にしすぎないため、追加テーブルは最小限とし、まずは既存テーブルへのカラム追加を中心に実装する。

## 2. 既存テーブル

初期版では、問い合わせ情報を `inquiries` テーブルで管理している。

主な既存カラムは以下。

| カラム名                | 内容               |
| ----------------------- | ------------------ |
| request_id              | 問い合わせID       |
| request_date            | 受付日             |
| request_time            | 受付時刻           |
| requester               | 依頼者             |
| department              | 部署               |
| channel                 | 受付経路           |
| category                | カテゴリ           |
| subcategory             | サブカテゴリ       |
| detail                  | 問い合わせ内容     |
| missing_info            | 不足情報           |
| priority                | 優先度             |
| due_date                | 希望期限           |
| assignee                | 担当者             |
| status                  | ステータス         |
| response_summary        | 対応内容           |
| record_issue            | 記録・管理上の問題 |
| completed_date          | 完了日             |
| management_minutes      | 管理作業時間       |
| actual_response_minutes | 実対応時間         |
| created_at              | 作成日時           |
| updated_at              | 更新日時           |

## 3. Ver.2追加カラム

Ver.2では、以下のカラムを追加する。

| カラム名               | 型      | 初期値 | 内容                        |
| ---------------------- | ------- | ------ | --------------------------- |
| faq_candidate          | INTEGER | 0      | FAQ候補フラグ。1ならFAQ候補 |
| faq_title              | TEXT    | 空文字 | FAQ化する場合のタイトル     |
| faq_answer             | TEXT    | 空文字 | FAQ回答案                   |
| additional_info        | TEXT    | 空文字 | カテゴリ別追加情報          |
| requester_visible      | INTEGER | 1      | 依頼者向け画面に表示するか  |
| last_status_changed_at | TEXT    | 空文字 | 最終ステータス変更日時      |

## 4. 追加カラムの詳細

### 4.1 `faq_candidate`

FAQ候補かどうかを示す。

| 値  | 意味            |
| --- | --------------- |
| 0   | FAQ候補ではない |
| 1   | FAQ候補である   |

原則として、完了済み問い合わせの中から管理部担当者が選択する。

### 4.2 `faq_title`

FAQとして整理する場合のタイトルを保存する。

例：

- 販売管理システムにログインできない場合の対応
- 共有フォルダの権限追加依頼方法
- 勤怠修正を依頼する場合の必要情報

### 4.3 `faq_answer`

FAQとして整理する場合の回答案を保存する。

問い合わせの `response_summary` をもとに、依頼者向けに分かりやすく書き換えた内容を保存する。

### 4.4 `additional_info`

カテゴリ別入力フォームで入力された追加情報を保存する。

Ver.2では、DBカラムをカテゴリごとに増やしすぎないため、追加情報を1つのテキスト項目として保存する。

保存形式は、読みやすさを優先して複数行テキストとする。

例：

```text
PC管理番号: PC-014
発生時刻: 2026-07-01 09:30
エラー内容: 販売管理システム起動時に認証エラー
再起動有無: 実施済み
```

### 4.5 `requester_visible`

依頼者向け確認画面に表示するかを示す。

| 値  | 意味                       |
| --- | -------------------------- |
| 0   | 依頼者向け画面に表示しない |
| 1   | 依頼者向け画面に表示する   |

Ver.2では、初期値は1とする。

ただし、管理部内部の確認事項など、依頼者に見せたくない問い合わせは0に変更できる設計とする。

### 4.6 `last_status_changed_at`

ステータスが最後に変更された日時を保存する。情報待ち長期化などの判定に利用する。

Ver.2初期実装では、取得が難しい場合は updated_at を代替的に利用してもよい。

## 5. 派生列

DBには保存せず、Python / pandas 側で計算する列は以下とする。

| 派生列                | 内容                 |
| --------------------- | -------------------- |
| has_alert             | アラート対象かどうか |
| has_alert_int         | アラート対象なら1    |
| alert_type            | アラート種別         |
| days_until_due        | 期限までの日数       |
| days_overdue          | 期限超過日数         |
| faq_candidate_int     | FAQ候補なら1         |
| requester_visible_int | 依頼者表示対象なら1  |

## 6. アラート判定条件

Ver.2のアラート判定条件は以下とする。

| アラート種別   | 条件                                                        |
| -------------- | ----------------------------------------------------------- |
| 期限超過       | `status != "完了"` かつ `due_date < 今日`                   |
| 本日期限       | `status != "完了"` かつ `due_date == 今日`                  |
| 期限間近       | `status != "完了"` かつ `due_date <= 今日 + 1日`            |
| 担当者未設定   | `assignee` が空                                             |
| 情報待ち長期化 | `status == "情報待ち"` かつ `request_date` から一定日数経過 |

情報待ち長期化の日数は、初期値として3日以上とする。

## 7. カテゴリ別追加情報

カテゴリ別入力フォームでは、以下の追加項目を扱う。

| カテゴリ         | 追加項目                                     |
| ---------------- | -------------------------------------------- |
| PC・システム     | PC管理番号、発生時刻、エラー内容、再起動有無 |
| アカウント・権限 | 対象システム、対象フォルダ、必要権限、承認者 |
| 勤怠・労務       | 対象日、修正前時刻、修正後時刻、理由         |
| 経費・請求       | 金額、取引先、対象月、添付書類有無           |
| 備品・設備       | 備品名、数量、利用目的、希望日               |

上記に該当しないカテゴリの場合は、自由記述の追加情報欄を表示する。

## 9. DB移行方針

既存DBにはVer.2追加カラムが存在しないため、移行処理を作成する。

移行処理では、以下を行う。

1. 既存の `inquiries` テーブル構造を確認する
2. 不足しているVer.2追加カラムを判定する
3. `ALTER TABLE` で不足カラムを追加する
4. 既存データには初期値を設定する

作成予定ファイル：

```text
src/migrate_db.py
```

## 10. データ設計上の注意点

Ver.2では、ポートフォリオとして分かりやすい実装を優先する。

そのため、以下の方針とする。

- カテゴリ別追加情報は、まずは `additional_info` にまとめて保存する
- FAQ候補は専用テーブルではなく、まずは `inquiries` テーブル上で管理する
- 本格的なFAQ公開ページはVer.3以降とする
- 本格的なユーザー管理・権限管理はVer.3以降とする
- SQLiteで扱いやすいシンプルな構造を維持する
