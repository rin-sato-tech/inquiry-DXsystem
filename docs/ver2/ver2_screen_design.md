# Ver.2 画面設計書

## 1. 画面設計の方針

Ver.2では、初期版の画面構成を大きく変更せず、既存画面を拡張しながら新しい機能を追加する。

初期版では、以下の4タブを実装していた。

- 問い合わせ一覧
- 新規登録
- ステータス更新
- 集計・CSV出力

Ver.2では、以下を追加・拡張する。

- 要対応アラート
- FAQ候補管理
- 依頼者向け確認
- 新規登録フォームのカテゴリ別入力項目
- 集計・CSV出力画面のVer.2対応

## 2. Ver.2の画面構成

Ver.2のタブ構成は以下とする。

| タブ           | 目的                                            |
| -------------- | ----------------------------------------------- |
| 要対応アラート | 期限超過・期限間近・担当者未設定などを確認する  |
| 問い合わせ一覧 | 全問い合わせを一覧・絞り込み表示する            |
| 新規登録       | 問い合わせを登録する                            |
| ステータス更新 | 担当者、ステータス、対応内容、FAQ候補を更新する |
| FAQ候補管理    | FAQ候補の一覧確認・編集・CSV出力を行う          |
| 依頼者向け確認 | 依頼者が自分の問い合わせ状況を確認する          |
| 集計・CSV出力  | 集計結果とTableau用CSV出力を確認する            |

## 3. 要対応アラート画面

### 3.1 目的

管理部担当者が、優先的に確認すべき問い合わせをすぐ把握できるようにする。

### 3.2 表示するKPI

- 期限超過件数
- 本日期限件数
- 期限間近件数
- 担当者未設定件数
- 情報待ち長期化件数

### 3.3 表示する一覧

以下の一覧を表示する。

- 期限超過問い合わせ
- 本日期限問い合わせ
- 期限間近問い合わせ
- 担当者未設定問い合わせ
- 情報待ち長期化問い合わせ

### 3.4 表示項目

| 項目         | 内容         |
| ------------ | ------------ |
| request_id   | 問い合わせID |
| request_date | 受付日       |
| requester    | 依頼者       |
| department   | 部署         |
| category     | カテゴリ     |
| priority     | 優先度       |
| due_date     | 希望期限     |
| assignee     | 担当者       |
| status       | ステータス   |
| alert_type   | アラート種別 |

## 4. 問い合わせ一覧画面

### 4.1 目的

全問い合わせを一覧表示し、条件で絞り込めるようにする。

### 4.2 Ver.2での追加要素

初期版の一覧表示に加え、以下の項目を表示候補に追加する。

- FAQ候補フラグ
- アラート有無
- アラート種別
- カテゴリ別追加情報

### 4.3 フィルタ追加案

既存フィルタに加え、以下を追加する。

- FAQ候補のみ
- アラートありのみ
- 依頼者表示対象のみ

## 5. 新規登録画面

### 5.1 目的

問い合わせを登録する。

### 5.2 Ver.2での追加要素

カテゴリ選択に応じて、追加入力項目を表示する。

### 5.3 カテゴリ別追加項目

| カテゴリ         | 追加項目                                     |
| ---------------- | -------------------------------------------- |
| PC・システム     | PC管理番号、発生時刻、エラー内容、再起動有無 |
| アカウント・権限 | 対象システム、対象フォルダ、必要権限、承認者 |
| 勤怠・労務       | 対象日、修正前時刻、修正後時刻、理由         |
| 経費・請求       | 金額、取引先、対象月、添付書類有無           |
| 備品・設備       | 備品名、数量、利用目的、希望日               |

### 5.4 保存方法

カテゴリ別追加項目は、まずは `additional_info` にまとめて保存する。

Ver.2ではDB構造を複雑にしすぎず、カテゴリ別の詳細情報を1つのテキスト項目に集約する。

## 6. ステータス更新画面

### 6.1 目的

問い合わせの担当者、ステータス、完了日、対応内容などを更新する。

### 6.2 Ver.2での追加要素

以下の項目を更新できるようにする。

- FAQ候補にするか
- FAQタイトル
- FAQ回答案
- 依頼者に表示するか
- 追加情報の確認

### 6.3 FAQ候補登録の条件

FAQ候補にできるのは、原則としてステータスが「完了」の問い合わせとする。

## 7. FAQ候補管理画面

### 7.1 目的

完了済み問い合わせの中からFAQ候補を管理し、問い合わせ削減に活用する。

### 7.2 機能

- FAQ候補一覧表示
- カテゴリ別FAQ候補件数表示
- FAQタイトル編集
- FAQ回答案編集
- FAQ候補CSV出力

### 7.3 表示項目

| 項目             | 内容           |
| ---------------- | -------------- |
| request_id       | 問い合わせID   |
| category         | カテゴリ       |
| subcategory      | サブカテゴリ   |
| detail           | 問い合わせ内容 |
| response_summary | 対応内容       |
| faq_title        | FAQタイトル    |
| faq_answer       | FAQ回答案      |
| completed_date   | 完了日         |

## 8. 依頼者向け確認画面

### 8.1 目的

依頼者が、自分の問い合わせ状況を確認できるようにする。

### 8.2 検索条件

- 問い合わせID
- 依頼者名

### 8.3 表示項目

依頼者向けには、管理用の内部情報を出しすぎないようにする。

| 項目                    | 表示       |
| ----------------------- | ---------- |
| request_id              | 表示する   |
| request_date            | 表示する   |
| requester               | 表示する   |
| department              | 表示する   |
| category                | 表示する   |
| detail                  | 表示する   |
| status                  | 表示する   |
| assignee                | 表示する   |
| due_date                | 表示する   |
| completed_date          | 表示する   |
| response_summary        | 表示する   |
| management_minutes      | 表示しない |
| actual_response_minutes | 表示しない |
| record_issue            | 表示しない |
| faq_answer              | 表示しない |

### 8.4 注意書き

画面上には以下の注意書きを表示する。

「この画面は、依頼者が自分の問い合わせ状況を確認する想定のデモ画面です。本格運用ではログイン認証・権限管理が必要です。」

## 9. 集計・CSV出力画面

### 9.1 目的

問い合わせ状況を集計し、Tableau用CSVを出力する。

### 9.2 Ver.2での追加集計

- アラートあり件数
- アラート種別別件数
- FAQ候補件数
- カテゴリ別FAQ候補件数
- 依頼者表示対象件数

### 9.3 Tableau出力に追加する項目

- faq_candidate
- faq_candidate_int
- alert_type
- has_alert
- has_alert_int
- additional_info
- requester_visible
- requester_visible_int
- last_status_changed_at

## 10. 画面設計上の注意点

Ver.2では機能が増えるため、画面が複雑になりすぎないようにする。

特に以下に注意する。

- タブを増やしすぎない
- 管理部向け画面と依頼者向け画面を混ぜすぎない
- FAQ候補管理と問い合わせ更新の役割を分ける
- 一覧表に表示する列を増やしすぎない
- 詳細情報は必要に応じて展開表示にする
