# Ver.3 アーキテクチャ設計書

## 1. 目的

本ドキュメントは、社内問い合わせ管理システム Ver.3 のアーキテクチャ方針を定義するものである。

Ver.3は、本格的な問い合わせ管理SaaSや社内業務システムを置き換えるものではない。本システムは、メール・Excel・口頭・チャットで分散していた社内問い合わせ対応を題材に、業務フロー整理、データ構造化、FAQ化、対応漏れ防止、履歴管理、集計・可視化を試作するポートフォリオ用PoCである。
そのため、Ver.3のアーキテクチャでは、以下を重視する。

| 方針                   | 内容                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| PoCとして成立させる    | 小規模な部門内業務改善アプリとして実装する                           |
| 既存構成を活かす       | Ver.2.0.1までのStreamlit + SQLite構成を継続する                      |
| 責務を整理する         | `app.py`, `src/ui/`, `src/services/`, `src/db.py` の役割を明確にする |
| 作り込みすぎない       | 本格認証、外部通知、添付、承認、完全監査ログは対象外にする           |
| 説明しやすくする       | 面接・ポートフォリオで設計意図を説明できる構成にする                 |
| 段階的拡張を可能にする | 将来、既製品導入・外注・内製継続を比較できる材料にする               |

---

## 2. Ver.3のアーキテクチャ上の位置づけ

Ver.3は、本番運用システムではなく、本格導入前のPoC拡張である。

| バージョン | アーキテクチャ上の位置づけ                                                         |
| ---------- | ---------------------------------------------------------------------------------- |
| Ver.1      | 問い合わせ登録・一覧・更新・集計の基本機能を持つ単体Streamlitアプリ                |
| Ver.2      | アラート、FAQ候補、依頼者確認、カテゴリ別入力、Tableau出力を追加した業務改善アプリ |
| WBS0       | `app.py` の肥大化を解消し、画面処理を `src/ui/` に分割                             |
| Ver.3      | 簡易ログイン、FAQ公開、依頼者画面、履歴管理、通知対象抽出を追加するPoC拡張         |
| 本格導入   | 既製品導入、外注開発、内製継続を比較検討する段階                                   |

Ver.3では、システムの完成度を本番運用レベルまで引き上げるのではなく、以下を示すことを目的とする。

- 業務課題を構造化できること
- 画面・データ・業務フローを設計できること
- 小規模な業務アプリとして動くものを実装できること
- 集計・BI出力までつなげられること
- どこまで内製し、どこから既製品・外注を検討すべきか判断できること

## 3. 技術構成

Ver.3では、Ver.2.0.1までの技術構成を継続する。

| 領域       | 技術                         | 方針                                     |
| ---------- | ---------------------------- | ---------------------------------------- |
| UI         | Streamlit                    | 継続利用する                             |
| 言語       | Python                       | 継続利用する                             |
| DB         | SQLite                       | 継続利用する                             |
| データ処理 | pandas                       | 集計・CSV出力で利用する                  |
| 可視化連携 | Tableau用CSV                 | CSV出力までを対象とする                  |
| テスト     | Pythonスクリプト、手動テスト | スモークテストとチェックリストで確認する |
| デプロイ   | ローカル実行前提             | クラウドデプロイは必須としない           |

Ver.3では、以下の技術移行は行わない。

| 行わないこと                | 理由                                                   |
| --------------------------- | ------------------------------------------------------ |
| PostgreSQL移行              | DB運用・インフラ寄りになり、PoCの主目的から外れるため  |
| Flask / Django / FastAPI化  | Streamlitで業務改善PoCの目的を満たせるため             |
| React等のフロントエンド導入 | 画面開発の複雑度が上がりすぎるため                     |
| Docker化                    | 必須ではない。必要に応じて将来課題とする               |
| クラウドデプロイ            | ポートフォリオではREADME、画面説明、実行手順を優先する |
| 外部通知API連携             | メール・Slack・Teams送信はVer.3対象外のため            |

## 4. 全体構成

Ver.3の全体構成は以下とする。

```text
app.py
  ↓
src/ui/
  ↓
src/services/
  ↓
src/db.py
  ↓
SQLite database
  ↓
CSV export
  ↓
Tableau
```

各層の役割は以下である。

| 層              | 主な役割                                                           |
| --------------- | ------------------------------------------------------------------ |
| `app.py`        | アプリ起動、初期化、データ読み込み、ログイン状態管理、画面呼び出し |
| `src/ui/`       | Streamlitによる画面表示、フォーム、ボタン、ユーザー操作            |
| `src/services/` | 業務ロジック、データ加工、通知文生成、ロール判定、履歴保存処理     |
| `src/db.py`     | SQLite接続、テーブル初期化、CRUD処理                               |
| SQLite          | 問い合わせ、利用者、FAQ、履歴、通知ログを保存                      |
| CSV             | Tableau連携用データ出力                                            |
| Tableau         | 集計結果の可視化                                                   |

## 5. WBS0で実施した構造整理

Ver.2までの `app.py` は、画面表示、フォーム処理、更新処理、集計表示などが集中しやすい構成だった。WBS0では、Ver.3拡張に備えて、画面単位で `src/ui/` に分割した。

| 分割前                           | 分割後                                          |
| -------------------------------- | ----------------------------------------------- |
| `app.py` に画面処理が集中        | 画面処理を `src/ui/` に分割                     |
| 画面追加ごとに `app.py` が肥大化 | `app.py` は入口・呼び出し中心にする             |
| 機能追加時の影響範囲が広い       | 画面単位で修正できる                            |
| Ver.3機能追加が難しい            | ログイン、FAQ公開、通知、履歴画面を追加しやすい |

WBS0後の基本方針は以下である。

| ファイル/ディレクトリ   | 役割                                                               |
| ----------------------- | ------------------------------------------------------------------ |
| `app.py`                | アプリの入口、DB初期化、キャッシュ付きデータ読み込み、画面呼び出し |
| `src/ui/`               | Streamlit画面                                                      |
| `src/services/`         | 業務ロジックの置き場                                               |
| `src/db.py`             | DB操作                                                             |
| `src/master_data.py`    | マスタデータ                                                       |
| `src/tableau_export.py` | Tableau用CSV出力                                                   |
| `src/smoke_test*.py`    | スモークテスト                                                     |

## 6. 実装区分

Ver.3では、アーキテクチャ上も以下の実装区分を採用する。

| 区分       | 内容                                              |
| ---------- | ------------------------------------------------- |
| 必須実装   | Ver.3の中心価値に直結する機能                     |
| 簡易実装   | 実務感を示すが、本格運用レベルまでは作らない機能  |
| 継続・拡張 | Ver.2までの機能を維持し、必要に応じて拡張するもの |
| 対象外     | Ver.3では作らないもの                             |

### 6.1 必須実装

| 領域           | 内容                                                       |
| -------------- | ---------------------------------------------------------- |
| 簡易ログイン   | ユーザー選択式ログイン                                     |
| ロール別表示   | requester / staff / admin / viewer の画面切替              |
| FAQ公開・検索  | 公開FAQの登録、検索、詳細表示                              |
| 依頼者画面     | 自分の問い合わせ一覧                                       |
| コメント履歴   | 問い合わせ単位のコメント保存                               |
| ステータス履歴 | ステータス変更前後の保存                                   |
| DB拡張         | `users`, `faq_items`, `inquiry_comments`, `status_history` |
| Ver.3テスト    | スモークテスト、手動テスト                                 |

### 6.2 簡易実装

| 領域            | 内容                                                 |
| --------------- | ---------------------------------------------------- |
| 操作履歴        | 主要操作のみ保存                                     |
| 通知対象抽出    | 期限前、期限超過、担当者未設定、情報待ち長期化を抽出 |
| 通知文生成      | 画面表示とログ保存まで                               |
| 通知ログ        | `created`, `reviewed`, `skipped` 程度                |
| Ver.3集計       | FAQ、通知、履歴の簡易集計                            |
| Tableau出力拡張 | Ver.3項目をCSVに追加                                 |

### 6.3 対象外

| 領域             | 内容                                   |
| ---------------- | -------------------------------------- |
| 本格認証         | パスワード、OAuth、社内認証連携        |
| 外部通知         | メール、Slack、Teams実送信             |
| 添付管理         | ファイルアップロード、保存、権限制御   |
| 承認ワークフロー | 稟議、承認、差戻し                     |
| 完全監査ログ     | 改ざん防止、アクセスログ、内部統制対応 |
| DB本格化         | PostgreSQL移行、高度な制約設計         |
| 本格Web化        | Flask / Django / FastAPI移行           |
| AI分類           | 問い合わせ自動分類、FAQ自動生成        |
| 本番運用対応     | 多人数同時利用、障害対応、運用監視     |

## 7. `app.py` の役割

Ver.3でも、`app.py` はアプリの入口として残す。`app.py` が担当するものは以下である。

| 役割                 | 内容                                            |
| -------------------- | ----------------------------------------------- |
| アプリ起動           | Streamlitアプリの入口                           |
| DB初期化             | `init_db()` を呼び出す                          |
| データ読み込み       | 問い合わせデータを読み込む                      |
| キャッシュ管理       | `@st.cache_data` によるデータ読み込みキャッシュ |
| ログイン状態確認     | `st.session_state` のユーザー情報を確認する     |
| ロール別メニュー生成 | ログイン中ロールに応じて表示画面を切り替える    |
| UI呼び出し           | `src/ui/` 配下の画面関数を呼び出す              |

`app.py` に書きすぎないものは以下である。

| 書かないもの         | 理由                                                |
| -------------------- | --------------------------------------------------- |
| 各画面の詳細表示処理 | `src/ui/` に分けるため                              |
| FAQ検索ロジック      | `src/services/faq_service.py` に分けるため          |
| 通知対象抽出ロジック | `src/services/notification_service.py` に分けるため |
| 履歴保存ロジック     | `src/services/history_service.py` に分けるため      |
| DB直接操作の詳細     | `src/db.py` に集約するため                          |

---

## 8. `src/ui/` の役割

`src/ui/` は、Streamlitの画面表示を担当する。

| 役割         | 内容                                           |
| ------------ | ---------------------------------------------- |
| 画面表示     | 見出し、説明文、表、フォーム、ボタンを表示する |
| 入力受付     | ユーザー入力を受け取る                         |
| 操作実行     | ボタン押下時にサービス層またはDB関数を呼び出す |
| エラー表示   | 入力不足や更新失敗を画面上に表示する           |
| ロール別表示 | 画面内で必要に応じて表示内容を切り替える       |

`src/ui/` に書きすぎないものは以下である。

| 書かないもの           | 理由                                                |
| ---------------------- | --------------------------------------------------- |
| 複雑な業務判定         | `src/services/` に分けるため                        |
| SQL処理                | `src/db.py` に集約するため                          |
| 通知文テンプレート生成 | `src/services/notification_service.py` に分けるため |
| 履歴保存ルール         | `src/services/history_service.py` に分けるため      |

## 9. `src/services/` の役割

`src/services/` は、Ver.3で追加される業務ロジックを配置する。
ただし、Ver.3はPoCであるため、過度に細かいレイヤー分割は行わない。必要な処理から段階的にサービス層へ分離する。

### 9.1 想定サービス一覧

| ファイル                  | 実装区分   | 役割                                           |
| ------------------------- | ---------- | ---------------------------------------------- |
| `auth_service.py`         | 必須       | 簡易ログイン、ロール判定、画面利用可否判定     |
| `faq_service.py`          | 必須       | FAQ検索、FAQ作成、公開・非公開切替、閲覧数加算 |
| `inquiry_service.py`      | 継続・拡張 | 問い合わせ登録・更新処理の補助                 |
| `history_service.py`      | 必須・簡易 | コメント履歴、ステータス履歴、操作履歴の保存   |
| `notification_service.py` | 簡易       | 通知対象抽出、通知文生成、通知ログ保存         |
| `reporting_service.py`    | 簡易       | Ver.3集計、Tableau出力用データ作成補助         |

### 9.2 サービス層に置く処理

| 処理                             | 配置先                                        |
| -------------------------------- | --------------------------------------------- |
| ロール別メニュー判定             | `auth_service.py`                             |
| 依頼者が見られる問い合わせの抽出 | `auth_service.py` または `inquiry_service.py` |
| 公開FAQ検索                      | `faq_service.py`                              |
| FAQ候補から公開FAQを作成         | `faq_service.py`                              |
| FAQ閲覧数・役立ち件数の加算      | `faq_service.py`                              |
| コメント保存                     | `history_service.py`                          |
| ステータス履歴保存               | `history_service.py`                          |
| 主要操作ログ保存                 | `history_service.py`                          |
| 期限前・期限超過の通知対象抽出   | `notification_service.py`                     |
| 通知文生成                       | `notification_service.py`                     |
| 通知ログ保存                     | `notification_service.py`                     |
| Ver.3集計                        | `reporting_service.py`                        |

### 9.3 サービス層で作り込まないもの

| 作らないもの         | 理由                                |
| -------------------- | ----------------------------------- |
| 認証基盤             | 本格認証は対象外のため              |
| 権限マスタエンジン   | `users.role` の簡易判定で十分なため |
| 通知送信基盤         | 外部送信しないため                  |
| 監査ログ基盤         | 完全監査ログは対象外のため          |
| ワークフローエンジン | 承認・差戻しは対象外のため          |

## 10. `src/db.py` の役割

`src/db.py` は、SQLiteへの接続、テーブル初期化、CRUD処理を担当する。
Ver.3でも、repository層の本格分離は必須としない。ポートフォリオとしては、`src/db.py` にDB操作をまとめ、サービス層から呼び出せる状態にする。

### 10.1 `src/db.py` に追加する主な処理

| 領域              | 処理                                                               |
| ----------------- | ------------------------------------------------------------------ |
| users             | ユーザー一覧取得、有効ユーザー取得、ユーザーID取得                 |
| faq_items         | FAQ作成、FAQ更新、公開FAQ取得、FAQ検索、閲覧数加算、役立ち件数加算 |
| inquiry_comments  | コメント追加、問い合わせ別コメント取得                             |
| status_history    | ステータス履歴追加、問い合わせ別履歴取得                           |
| operation_logs    | 操作ログ追加、操作ログ一覧取得                                     |
| notification_logs | 通知ログ追加、通知ログ一覧取得、状態更新                           |
| reports           | Ver.3集計用データ取得                                              |

### 10.2 `src/db.py` に入れすぎないもの

| 入れすぎないもの      | 理由                                                |
| --------------------- | --------------------------------------------------- |
| 画面表示処理          | `src/ui/` の責務であるため                          |
| 通知文生成ロジック    | `notification_service.py` の責務であるため          |
| ロール別画面判定      | `auth_service.py` の責務であるため                  |
| FAQ公開可否の業務判断 | `faq_service.py` またはUI側の管理者操作に分けるため |
| 集計結果の画面整形    | `src/ui/report_page.py` の責務であるため            |

## 11. データベースアーキテクチャ

Ver.3では、既存の `inquiries` テーブルを問い合わせ本体として継続利用する。そのうえで、Ver.3追加機能に必要なテーブルを追加する。

| テーブル            | 実装区分 | 役割                           |
| ------------------- | -------- | ------------------------------ |
| `inquiries`         | 継続     | 問い合わせ本体                 |
| `users`             | 必須     | 簡易ログイン、ロール管理       |
| `faq_items`         | 必須     | 公開FAQ管理                    |
| `inquiry_comments`  | 必須     | 問い合わせコメント履歴         |
| `status_history`    | 必須     | ステータス変更履歴             |
| `operation_logs`    | 簡易     | 主要操作の簡易履歴             |
| `notification_logs` | 簡易     | 通知対象・通知文生成の簡易履歴 |

### 11.1 必須テーブル

| テーブル           | 理由                                   |
| ------------------ | -------------------------------------- |
| `users`            | ログインユーザーとロールを区別するため |
| `faq_items`        | FAQ候補を公開FAQとして独立管理するため |
| `inquiry_comments` | 対応経緯を時系列で残すため             |
| `status_history`   | ステータス変更の経緯を追うため         |

### 11.2 簡易テーブル

| テーブル            | 扱い                                               |
| ------------------- | -------------------------------------------------- |
| `operation_logs`    | 主要操作のみを保存する。完全監査ログではない       |
| `notification_logs` | 通知対象と通知文を保存する。通知送信キューではない |

### 11.3 作らないテーブル

| 作らないテーブル     | 理由                                       |
| -------------------- | ------------------------------------------ |
| `roles`              | `users.role` で十分なため                  |
| `user_roles`         | 1ユーザー1ロールの簡易設計で十分なため     |
| `attachments`        | 添付ファイル管理は対象外のため             |
| `faq_view_logs`      | `view_count`, `helpful_count` で十分なため |
| `notification_queue` | 外部送信を行わないため                     |
| `approval_steps`     | 承認ワークフローは対象外のため             |
| `access_logs`        | 本格監査ログは対象外のため                 |

## 12. キャッシュ方針

Ver.3でも、Streamlitの `@st.cache_data` を利用する。

### 12.1 キャッシュ対象

| 対象           | 方針                                 |
| -------------- | ------------------------------------ |
| 問い合わせ一覧 | 読み込み頻度が高いためキャッシュ対象 |
| マスタデータ   | 変更頻度が低いためキャッシュ対象     |
| 集計用データ   | 表示用にキャッシュ可能               |
| 公開FAQ一覧    | 必要に応じてキャッシュ可能           |

### 12.2 キャッシュクリア

DB更新後は、キャッシュをクリアして画面に最新データを反映する。

| 更新処理         | キャッシュクリア要否                         |
| ---------------- | -------------------------------------------- |
| 問い合わせ登録   | 必要                                         |
| 問い合わせ更新   | 必要                                         |
| FAQ作成・更新    | 必要                                         |
| FAQ公開・非公開  | 必要                                         |
| コメント追加     | 必要                                         |
| ステータス変更   | 必要                                         |
| 通知ログ保存     | 必要に応じて                                 |
| 操作ログ保存のみ | 原則不要。ただし履歴画面表示時は必要に応じて |

### 12.3 共通関数

キャッシュクリアは、`src/ui/cache_utils.py` に集約する。

```python
import streamlit as st

def clear_cache() -> None:
    st.cache_data.clear()
```

各画面ファイルでは、以下のように利用する。

```python
from src.ui.cache_utils import clear_cache
```

`from app import clear_cache` は循環importの原因になるため使用しない。

## 13. 簡易ログイン・ロール管理アーキテクチャ

### 13.1 方針

Ver.3では、ユーザー選択式の簡易ログインを行う。本格的なパスワード認証、OAuth、社内認証連携は実装しない。

### 13.2 ログイン状態

ログイン状態は `st.session_state` に保持する。

| キー                      | 内容                 |
| ------------------------- | -------------------- |
| `current_user_id`         | ログイン中ユーザーID |
| `current_user_name`       | ログイン中ユーザー名 |
| `current_user_role`       | ログイン中ロール     |
| `current_user_department` | ログイン中部署       |

### 13.3 ロール判定

ロールは `users.role` を参照して判定する。

| ロール      | 利用できる主な機能                            |
| ----------- | --------------------------------------------- |
| `requester` | FAQ検索、新規問い合わせ、自分の問い合わせ一覧 |
| `staff`     | 問い合わせ一覧、更新、アラート、FAQ候補登録   |
| `admin`     | 全機能                                        |
| `viewer`    | 集計・CSV出力                                 |

### 13.4 作らないもの

| 作らないもの           | 理由                      |

## 15. 依頼者向け機能アーキテクチャ

### 15.1 方針

依頼者向け機能では、以下を実現する。

| 機能                   | 内容                                        |
| ---------------------- | ------------------------------------------- |
| FAQ検索                | 問い合わせ前の自己解決                      |

| 自分の問い合わせ一覧   | 登録済み問い合わせの進捗確認                |
| 依頼者向けコメント表示 | `visibility = requester` のコメントのみ表示 |

### 15.2 表示制御

依頼者には、以下を表示しない。

| 非表示情報           | 理由                         |

| 操作履歴             | 管理部内部向け情報のため     |
| ステータス履歴の詳細 | 内部対応経緯であるため       |
| 非公開FAQ            | 管理者確認前の情報であるため |
| 通知対象確認情報     | 管理部側の運用情報であるため |

### 15.3 作り込まないもの

| 作らないもの               | 理由                                 |
| -------------------------- | ------------------------------------ |
| 本格マイページ             | 自分の問い合わせ確認までで十分なため |
| 依頼者による問い合わせ編集 | 権限設計が複雑になるため             |
| 依頼者チャット             | コメント履歴は簡易メモとして扱うため |
| 添付ファイル提出           | Ver.3対象外のため                    |

## 16. 履歴管理アーキテクチャ

### 16.1 方針

履歴管理は、以下の3種類に分ける。

| 履歴           | 実装区分 | テーブル           | 扱い                 |
| -------------- | -------- | ------------------ | -------------------- |
| コメント履歴   | 必須     | `inquiry_comments` | 対応経緯を残す       |
| ステータス履歴 | 必須     | `status_history`   | 状態変更の経緯を残す |
| 操作履歴       | 簡易     | `operation_logs`   | 主要操作のみ保存する |

### 16.2 コメント履歴

コメント履歴は、問い合わせ対応の途中経過や補足情報を残すために使う。

| 項目        | 内容               |
| ----------- | ------------------ |
| `internal`  | 管理部内部向け     |
| `requester` | 依頼者にも表示可能 |

コメント履歴はチャット機能ではない。あくまで対応メモとして扱う。

### 16.3 ステータス履歴

ステータス変更時に、変更前後の状態を保存する。

| 保存項目         | 内容             |
| ---------------- | ---------------- |
| 問い合わせID     | 対象問い合わせ   |
| 変更前ステータス | 変更前の状態     |
| 変更後ステータス | 変更後の状態     |
| 変更者           | 操作したユーザー |
| 変更日時         | 操作日時         |

### 16.4 操作履歴

操作履歴は、主要操作のみを簡易保存する。

| 記録する操作    | 記録しない操作       |
| --------------- | -------------------- |
| 問い合わせ作成  | 画面閲覧             |
| 問い合わせ更新  | ログイン・ログアウト |
| ステータス変更  | FAQ閲覧の個別ログ    |
| 担当者変更      | すべての入力変更差分 |
| FAQ作成・更新   | アクセスログ         |
| FAQ公開・非公開 | -                    |
| コメント追加    | -                    |
| 通知文生成      | -                    |

### 16.5 作り込まないもの

| 作らないもの             | 理由                                |
| ------------------------ | ----------------------------------- |
| 完全監査ログ             | 内部統制システムではないため        |
| アクセスログ             | 本格監査対象外のため                |
| 差分詳細の完全保存       | 実装量が大きく、PoC範囲を超えるため |
| ログ改ざん防止           | 本格監査基盤ではないため            |
| ログエクスポート専用機能 | Ver.3では不要なため                 |

## 17. 通知対象抽出アーキテクチャ

### 17.1 方針

Ver.3では、外部通知の実送信は行わない。通知対象抽出、通知文生成、通知ログ保存までを対象とする。
通知機能は、本格通知基盤ではなく、対応漏れ防止のPoCとして扱う。

### 17.2 データの流れ

```text
inquiries
  ↓ 条件抽出
notification_service.py
  ↓ 通知文生成
notification_logs
  ↓ 画面確認
notification_page.py
```

### 17.3 通知種別

| 通知種別           | 条件例                       | 通知先の考え方 |
| ------------------ | ---------------------------- | -------------- |
| `before_due`       | 期限が近い未完了問い合わせ   | 担当者         |
| `overdue`          | 期限を過ぎた未完了問い合わせ | 担当者、管理者 |
| `unassigned`       | 担当者が未設定               | 管理者         |
| `waiting_too_long` | 情報待ち状態が長期化         | 担当者         |
| `status_changed`   | ステータスが変更された       | 依頼者         |
| `completed`        | 問い合わせが完了した         | 依頼者         |

### 17.4 通知状態

| 状態       | 意味                         |
| ---------- | ---------------------------- |
| `created`  | 通知文生成済み               |
| `reviewed` | 管理者または担当者が確認済み |
| `skipped`  | 通知対象から除外             |

Ver.3では、`sent`, `failed`, `retry` は使用しない。外部送信を行わないためである。

### 17.5 作り込まないもの

| 作らないもの             | 理由                                  |
| ------------------------ | ------------------------------------- |
| メール送信               | 外部通知は対象外のため                |
| Slack / Teams送信        | 同上                                  |
| 通知キュー               | 実送信しないため不要                  |
| 再送制御                 | 実送信しないため不要                  |
| 送信失敗管理             | 実送信しないため不要                  |
| 通知テンプレート管理画面 | PoCでは関数内テンプレートで十分なため |

## 22. エラーハンドリング方針

Ver.3では、PoCとして最低限のエラーハンドリングを行う。

| 場面            | 方針                                             |
| --------------- | ------------------------------------------------ |
| 必須入力不足    | 画面上にエラー表示し、DB登録しない               |
| DB更新失敗      | エラー表示し、更新処理を中断する                 |
| 権限外操作      | 利用不可メッセージを表示する                     |
| 対象データなし  | 空表示ではなく、対象なしメッセージを表示する     |

| 通知対象なし    | 通知対象なしと表示し、ログを作成しない           |

本格的な例外監視、ログ収集、障害通知はVer.3対象外とする。

## 26. 実運用時の導入判断

Ver.3はPoCであり、本格運用に進む場合は以下を比較する。

| 選択肢       | 向いている状況                                            |
| ------------ | --------------------------------------------------------- |
| 内製継続     | 小規模部署内で利用し、要件が限定的な場合                  |
| 既製品導入   | 複数部署・全社利用、認証、通知、添付、承認が必要な場合    |
| 外注開発     | 自社固有の業務フローが強く、既製品では対応しにくい場合    |
| 現行運用改善 | システム化より、FAQ整備や業務ルール変更の方が効果的な場合 |

Ver.3のアーキテクチャは、この判断の前段階として、必要機能と改善効果を可視化するための構成である。

| 通知方針     | 外部送信せず、通知対象抽出・通知文生成・ログ保存までと整理されている                         |
| 集計方針     | Ver.3指標を簡易集計し、CSV出力までを対象としている                                           |
| 対象外       | 本格認証、外部通知、添付、承認、完全監査ログ、PostgreSQL移行などが対象外として整理されている |
| 将来拡張     | 本格導入時の検討領域が整理されている                                                         |

## 29. 結論

Ver.3のアーキテクチャでは、Ver.2.0.1までのStreamlit + SQLite構成を継続しつつ、WBS0で分割した `src/ui/` 構成を前提に、簡易ログイン、ロール別表示、FAQ公開・検索、依頼者画面、コメント履歴、ステータス履歴を中心に拡張する。
`users`, `faq_items`, `inquiry_comments`, `status_history` はVer.3の中心機能を支える必須テーブルとして扱う。一方で、`operation_logs` と `notification_logs` は、本格監査・通知基盤ではなく、主要操作や通知対象を確認するための簡易実装として扱う。
アプリケーション構造としては、`app.py` を入口とし、画面処理を `src/ui/`、業務ロジックを `src/services/`、DB操作を `src/db.py` に分ける。ただし、PoCであるため、過度なレイヤー分割や本格Webアプリ化は行わない。
Ver.3で作らないものは、本格認証、外部通知、添付ファイル、承認ワークフロー、完全な監査ログ、PostgreSQL移行、Flask / Django / FastAPI化、AI分類である。これらは、本格導入時に既製品導入・外注開発・内製継続を比較する領域として扱う。
この構成により、Ver.3は「本格システムを作り切る」のではなく、「業務改善PoCとして、依頼者の自己解決、対応経緯の可視化、対応漏れ候補の把握、改善分析、導入判断を説明できる状態」を目指す。
