# Ver.3 アーキテクチャ設計書

## 1. 目的

本ドキュメントは、社内問い合わせ管理システム Ver.3 におけるアプリケーション構成、責務分離方針、ファイル構成、実装ルールを整理するものである。

Ver.2.0.1までは、問い合わせ管理、要対応アラート、FAQ候補管理、カテゴリ別入力フォーム、依頼者向け確認、集計・CSV出力を実装した。
WBS0では、Ver.3本体開発に入る前の準備として、`app.py` に集中していた画面処理を `src/ui/` 配下へ分離した。

Ver.3では、このWBS0後の構造を前提に、簡易ログイン、ロール管理、FAQ公開、通知対象抽出、履歴管理、Ver.3集計を追加する。

---

## 2. アーキテクチャ設計の背景

Ver.2までの開発では、機能追加を優先したため、`app.py` に以下の処理が集中していた。

| 種類           | 内容                                                  |
| -------------- | ----------------------------------------------------- |
| 画面表示       | 一覧、登録、更新、アラート、FAQ候補、依頼者確認、集計 |
| 入力フォーム   | 新規問い合わせ、ステータス更新、FAQ候補登録           |
| DB更新呼び出し | 問い合わせ登録、問い合わせ更新、FAQ候補更新           |
| 集計表示       | 基本集計、Ver.2追加集計、Tableau CSV出力              |
| 共通処理       | キャッシュクリア、補助表示関数                        |

Ver.2までであればこの構成でも運用できるが、Ver.3では以下の機能が追加される。

| Ver.3追加要素  | 複雑化する理由                             |
| -------------- | ------------------------------------------ |
| 簡易ログイン   | ログインユーザー情報を各画面で利用する     |
| ロール管理     | ロールごとに表示画面・操作範囲を切り替える |
| FAQ公開        | FAQ候補と公開FAQを分けて扱う               |
| 通知対象抽出   | アラート情報から通知対象と通知文を生成する |
| 操作履歴       | 更新処理ごとに履歴保存が必要になる         |
| ステータス履歴 | ステータス変更時に履歴保存が必要になる     |
| コメント履歴   | 問い合わせ単位の時系列メモが必要になる     |
| Ver.3集計      | FAQ、通知、履歴に関する集計が増える        |

そのため、Ver.3では画面表示、業務ロジック、DB操作の責務を整理し、機能追加に耐えられる構成にする。

## 4. Ver.3での責務分離方針

Ver.3では、以下の責務分離を基本とする。

| 層           | 役割                                                 | 主な配置先                                                      |
| ------------ | ---------------------------------------------------- | --------------------------------------------------------------- |
| アプリ入口   | 初期化、データ読み込み、タブ構成、画面呼び出し       | `app.py`                                                        |
| UI層         | Streamlit画面、フォーム、ボタン、表示制御            | `src/ui/`                                                       |
| サービス層   | 業務ロジック、登録・更新の流れ、権限制御、通知文生成 | `src/services/`                                                 |
| DB層         | SQLite接続、SQL実行、データ取得・更新                | `src/db.py`                                                     |
| 集計・分析層 | 集計、派生列作成、Tableau用データ作成                | `src/aggregation.py`, `src/summary.py`, `src/tableau_export.py` |
| ドメイン補助 | アラート判定、FAQ候補処理、依頼者表示処理            | `src/alerts.py`, `src/faq.py`, `src/requester_view.py`          |

## 5. `app.py` の役割

Ver.3における `app.py` は、アプリケーションの入口として扱う。

### 5.1 `app.py` が担当すること

| 担当           | 内容                                                    |
| -------------- | ------------------------------------------------------- |
| 初期化         | `init_db()` を呼び出す                                  |
| データ読み込み | `load_inquiries()` で問い合わせデータを読み込む         |
| キャッシュ     | `@st.cache_data` により問い合わせデータをキャッシュする |
| 画面構成       | タブまたはメニューを構成する                            |
| 画面呼び出し   | `src/ui/` の各画面関数を呼び出す                        |

### 5.2 `app.py` が担当しないこと

| 非担当                 | 理由                                                  |
| ---------------------- | ----------------------------------------------------- |
| 画面本体の詳細実装     | `src/ui/` に分離するため                              |
| 入力フォームの詳細処理 | 各UIページに配置するため                              |
| DB更新処理の詳細       | `src/db.py` または `src/services/` から呼び出すため   |
| 権限判定の詳細         | `src/services/auth_service.py` へ分離するため         |
| 通知文生成             | `src/services/notification_service.py` へ分離するため |
| 履歴保存処理           | `src/services/history_service.py` へ分離するため      |

### 5.3 WBS0時点の `load_inquiries()` の扱い

`load_inquiries()` は、WBS0時点では `app.py` に残してよい。
理由は、`@st.cache_data` がStreamlit固有の機能であり、無理にサービス層へ移すとサービス層がUIフレームワークに依存するためである。
将来的には、以下のように分離することを検討する。

```text
app.py
  - @st.cache_data付きの load_inquiries()

src/services/inquiry_service.py
  - get_inquiry_dataframe()
```

この場合、Streamlitキャッシュは `app.py` に残し、データ取得・加工処理だけをサービス層へ移す。

## 7. `src/services/` の役割

`src/services/` は、Ver.3本体で追加される業務ロジックを担当する。

### 7.1 サービス層を設ける理由

Ver.3では、単純なDB更新だけでなく、以下のような複数処理を組み合わせる必要がある。

| 処理           | 内容                                                                         |
| -------------- | ---------------------------------------------------------------------------- |
| 問い合わせ作成 | `inquiries` へ登録し、操作履歴・初期ステータス履歴を保存する                 |
| ステータス更新 | `inquiries.status` を更新し、`status_history` と `operation_logs` に保存する |
| FAQ公開        | `faq_items` を作成・更新し、操作履歴を保存する                               |
| 通知文生成     | 対象問い合わせを抽出し、通知文を作り、通知ログに保存する                     |
| ロール制御     | ログインユーザーのロールに応じて利用可能画面を判定する                       |

これらをUIファイルに直接書くと、画面が再び肥大化する。そのため、Ver.3では業務ロジックをサービス層に分離する。

### 7.2 Ver.3で追加するサービス候補

| ファイル                               | 役割                                         |
| -------------------------------------- | -------------------------------------------- |
| `src/services/auth_service.py`         | 簡易ログイン、ロール判定、メニュー制御       |
| `src/services/inquiry_service.py`      | 問い合わせ作成・更新の業務処理               |
| `src/services/faq_service.py`          | FAQ公開、FAQ検索、閲覧数・役立ち件数更新     |
| `src/services/notification_service.py` | 通知対象抽出、通知文生成、通知ログ保存       |
| `src/services/history_service.py`      | 操作履歴、ステータス履歴、コメント履歴の保存 |
| `src/services/reporting_service.py`    | Ver.3集計、Tableau出力用データ作成           |

### 7.3 サービス層の実装ルール

| ルール             | 内容                                                       |
| ------------------ | ---------------------------------------------------------- |
| UI表示を書かない   | `st.write`, `st.dataframe`, `st.button` などを原則書かない |
| 業務処理をまとめる | 複数テーブル更新や履歴保存をまとめる                       |
| DB関数を呼ぶ       | `src/db.py` の関数を利用してDBを操作する                   |
| 戻り値を明確にする | UI側で表示しやすい結果を返す                               |
| 例外処理を整理する | 失敗時の理由をUI側に返せるようにする                       |

## 8. `src/db.py` の扱い

Ver.3では、DB操作は当面 `src/db.py` に追加する。いきなり `repositories/` へ分離すると変更範囲が大きくなるため、Ver.3では段階的な整理を優先する。

### 8.1 `src/db.py` が担当すること

| 担当                  | 内容                                              |
| --------------------- | ------------------------------------------------- |
| DB接続                | SQLiteへの接続                                    |
| DB初期化              | `schema.sql` の実行                               |
| 問い合わせ取得        | `inquiries` からのデータ取得                      |
| 問い合わせ更新        | `inquiries` へのINSERT/UPDATE                     |
| Ver.3追加テーブル操作 | `users`, `faq_items`, `status_history` などのCRUD |

### 8.2 `src/db.py` の肥大化リスク

Ver.3でテーブルが増えるため、`src/db.py` が肥大化する可能性がある。

| リスク             | 内容                                              |
| ------------------ | ------------------------------------------------- |
| SQLが増える        | テーブル追加により関数数が増える                  |
| 役割が混在する     | 問い合わせ、FAQ、履歴、通知が同一ファイルに集まる |
| テストしにくくなる | 関数の責務が曖昧になる                            |

ただし、Ver.3では実装範囲を広げすぎないため、まずは `src/db.py` に追加し、後続バージョンで分割を検討する。

### 8.3 将来的なrepository分離案

将来的には、以下のような分割を検討する。

```text
src/repositories/
├── inquiry_repository.py
├── user_repository.py
├── faq_repository.py
├── history_repository.py
├── notification_repository.py
└── reporting_repository.py
```

Ver.3では、上記の構想を文書化するが、実装は必須としない。

## 9. キャッシュ処理方針

Ver.3でも、問い合わせ一覧データの読み込みには `st.cache_data` を利用する。

### 9.1 キャッシュを使う目的

| 目的              | 内容                                     |
| ----------------- | ---------------------------------------- |
| 表示速度向上      | 毎回DBを読み込まないようにする           |
| 画面操作の安定化  | 同一セッション内でデータ表示を安定させる |
| Streamlitとの相性 | Streamlitアプリとして自然な構成にする    |

### 9.2 キャッシュクリア方針

DB更新後は、共通関数 `clear_cache()` を呼び出してキャッシュを削除する。

```text
src/ui/cache_utils.py
  - clear_cache()
```

`clear_cache()` は、以下の処理後に呼ぶ。

| 処理                | キャッシュクリア                       |
| ------------------- | -------------------------------------- |
| 問い合わせ新規登録  | 必要                                   |
| 問い合わせ更新      | 必要                                   |
| FAQ候補更新         | 必要                                   |
| FAQ公開・非公開変更 | 必要                                   |
| コメント追加        | 必要                                   |
| 通知ログ保存        | 必要                                   |
| 履歴保存のみ        | 原則不要だが、表示に反映する場合は必要 |

### 9.3 避けるべき実装

以下は避ける。

```python
from app import clear_cache
```

理由は、`app.py` がUIファイルをimportし、UIファイルが `app.py` をimportする循環importが発生するためである。

## 10. ロール管理アーキテクチャ

Ver.3では、本格的な認証機能ではなく、ユーザー選択式の簡易ログインを採用する。

### 10.1 ログイン状態の保持

ログイン状態は、`st.session_state` に保持する。

| session_stateキー | 内容                       |
| ----------------- | -------------------------- |
| `current_user`    | ログイン中ユーザー情報     |
| `current_role`    | ログイン中ユーザーのロール |
| `is_logged_in`    | ログイン状態               |

### 10.2 ロール別メニュー制御

ロール別メニューは、`auth_service.py` に定義する。

```text
src/services/auth_service.py
  - get_current_user()
  - login_user()
  - logout_user()
  - get_available_pages()
  - has_permission()
```

UI側では、`get_available_pages()` の結果に応じてメニューを表示する。

### 10.3 ロール一覧

| ロール      | 主な利用画面                                            |
| ----------- | ------------------------------------------------------- |
| `requester` | 依頼者トップ、FAQ検索、新規問い合わせ、自分の問い合わせ |
| `staff`     | 問い合わせ一覧、更新、要対応アラート、FAQ候補管理       |
| `admin`     | 全画面                                                  |
| `viewer`    | 集計・CSV出力                                           |

## 11. FAQ機能アーキテクチャ

Ver.3では、Ver.2のFAQ候補とVer.3の公開FAQを分ける。

### 11.1 FAQ候補と公開FAQの違い

| 種類    | 保存先      | 内容                                  |
| ------- | ----------- | ------------------------------------- |
| FAQ候補 | `inquiries` | 完了済み問い合わせから抽出したFAQ候補 |
| 公開FAQ | `faq_items` | 依頼者向けに整備・公開するFAQ         |

### 11.2 FAQ公開処理の流れ

```text
FAQ候補選択
  ↓
faq_items 作成
  ↓
管理者が内容確認・編集
  ↓
公開フラグ更新
  ↓
operation_logs 保存
```

### 11.3 関連ファイル

| ファイル                      | 役割                                |
| ----------------------------- | ----------------------------------- |
| `src/ui/faq_admin_page.py`    | FAQ候補管理・公開操作UI             |
| `src/ui/faq_public_page.py`   | 依頼者向けFAQ検索・詳細UI           |
| `src/services/faq_service.py` | FAQ作成、検索、公開切替、閲覧数更新 |
| `src/db.py`                   | FAQ関連DB操作                       |
| `src/faq.py`                  | 既存FAQ候補処理の補助               |

対象問い合わせ抽出

通知文生成
  ↓

## 13. 履歴管理アーキテクチャ

Ver.3では、操作履歴、ステータス履歴、コメント履歴を追加する。

### 13.1 履歴の種類

| 履歴           | 保存先             | 内容                       |
| -------------- | ------------------ | -------------------------- |
| 操作履歴       | `operation_logs`   | 誰が、いつ、何を操作したか |
| ステータス履歴 | `status_history`   | ステータス変更の前後       |
| コメント履歴   | `inquiry_comments` | 対応中のメモ・コメント     |

### 13.2 履歴保存の基本方針

更新処理では、UIから直接複数のDB更新を呼ぶのではなく、サービス層の関数にまとめる。

例：

```text
UI
  ↓
inquiry_service.update_inquiry_status()
  ↓
inquiries 更新
status_history 保存
operation_logs 保存
```

この構成により、履歴登録漏れを防ぐ。

### 13.3 関連ファイル

| ファイル                          | 役割                             |
| --------------------------------- | -------------------------------- |
| `src/ui/history_page.py`          | 履歴確認画面                     |
| `src/services/history_service.py` | 履歴保存・取得の業務処理         |
| `src/services/inquiry_service.py` | 問い合わせ更新時に履歴保存を呼ぶ |
| `src/db.py`                       | 履歴テーブルへのINSERT/SELECT    |

## 14. 集計・Tableau出力アーキテクチャ

Ver.3では、Ver.1・Ver.2の集計に加えて、FAQ、通知、履歴に関する集計を追加する。

### 14.1 集計対象

| 分類      | 主な指標                                        |
| --------- | ----------------------------------------------- |
| 既存集計  | 総件数、未完了件数、期限超過件数、対応時間      |
| Ver.2集計 | FAQ候補件数、追加情報入力率、依頼者向け表示件数 |
| FAQ集計   | 公開FAQ件数、閲覧数、役立ち件数                 |
| 通知集計  | 通知種別別件数                                  |

## 16. Ver.3での実装ルール

Ver.3では、以下の実装ルールを守る。

### 16.1 UI層のルール

| ルール                    | 内容                                                 |
| ------------------------- | ---------------------------------------------------- |
| Streamlit処理はUI層に置く | `st.*` の処理は原則 `src/ui/` に置く                 |
| 複雑な業務処理を書かない  | 複数テーブル更新はサービス層に任せる                 |
| DB関数を直接呼びすぎない  | 可能な範囲でサービス層経由にする                     |
| 表示と処理を分ける        | 入力値取得、サービス呼び出し、結果表示を明確に分ける |

### 16.2 サービス層のルール

| ルール                | 内容                                                   |
| --------------------- | ------------------------------------------------------ |
| Streamlit依存を避ける | 原則 `st.*` を書かない                                 |
| 業務処理をまとめる    | DB更新、履歴保存、通知文生成などを一連の処理として扱う |
| 戻り値を明確にする    | 成功、失敗、メッセージ、生成IDなどを返す               |
| 履歴保存漏れを防ぐ    | 更新系処理では履歴保存をサービス内で呼ぶ               |

### 16.3 DB層のルール

| ルール                             | 内容                                                          |
| ---------------------------------- | ------------------------------------------------------------- |
| SQLを集約する                      | SQL文は原則 `src/db.py` に置く                                |
| UIにSQLを書かない                  | 画面ファイルにSQLを直接書かない                               |
| 追加テーブルごとに関数名を整理する | `insert_faq_item`, `fetch_users`, `insert_operation_log` など |
| 既存関数を壊さない                 | Ver.2.0.1のスモークテストが通る状態を維持する                 |

## 20. 結論

Ver.3では、WBS0で整理した `app.py` と `src/ui/` の構成を前提に、簡易ログイン、ロール管理、FAQ公開、通知対象抽出、履歴管理、Ver.3集計を追加する。

`app.py` はアプリの入口として、初期化、データ読み込み、画面呼び出しを担当する。
画面表示は `src/ui/` に配置し、業務ロジックは `src/services/` に段階的に分離する。
DB操作は当面 `src/db.py` を継続利用し、Ver.3後半または後続バージョンでrepository分離を検討する。

この構成により、Ver.2.0.1の機能を維持しながら、Ver.3で必要となる複数テーブル・ロール別表示・履歴保存・通知対象抽出に対応できる。
最終的には、問い合わせ管理アプリを「画面中心の個人開発アプリ」から、「責務分離された小規模業務システム」に近づけることを目指す。
