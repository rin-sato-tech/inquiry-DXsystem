# Ver.3 WBS

## 1. 目的

本ドキュメントは、社内問い合わせ管理システム Ver.3 の開発作業をWBSとして整理するものである。

Ver.3は、本格的な問い合わせ管理SaaSや社内業務システムを置き換えるものではない。本システムは、メール・Excel・口頭・チャットで分散していた社内問い合わせ対応を題材に、業務フロー整理、データ構造化、FAQ化、対応漏れ防止、履歴管理、集計・可視化を試作するポートフォリオ用PoCである。

そのため、Ver.3のWBSでは、すべての機能を同じ深さで作り込むのではなく、以下の区分で実装する。

| 区分       | 方針                                              |
| ---------- | ------------------------------------------------- |
| 必須実装   | Ver.3の中心価値に直結するため、優先的に実装する   |
| 簡易実装   | 実務感を示すために最低限実装する                  |
| 継続・拡張 | Ver.2.0.1までの機能を壊さず、必要に応じて拡張する |
| 対象外     | Ver.3では作らない                                 |

Ver.3の中心は、以下である。

- 簡易ログイン
- ロール別表示
- FAQ公開・検索
- 依頼者向け画面
- コメント履歴
- ステータス履歴
- 通知対象抽出の簡易実装
- Ver.3簡易集計
- Tableau用CSV出力拡張

一方で、本格認証、外部通知、添付ファイル、承認ワークフロー、完全監査ログ、PostgreSQL移行、Flask/Django/FastAPI化、AI分類はVer.3では対象外とする。

---

## 2. Ver.3開発方針

Ver.3では、Ver.2.0.1までの成果を壊さず、段階的に拡張する。

| 方針               | 内容                                                                               |
| ------------------ | ---------------------------------------------------------------------------------- |
| 既存機能維持       | Ver.2.0.1の問い合わせ登録、更新、アラート、FAQ候補、集計、CSV出力を維持する        |
| PoCとして実装      | 本番運用レベルではなく、業務改善の試作として実装する                               |
| 必須機能を優先     | FAQ公開、依頼者画面、コメント履歴、ステータス履歴を優先する                        |
| 簡易機能は軽く作る | 操作履歴、通知ログ、Ver.3集計は作り込みすぎない                                    |
| 本格機能は対象外   | 認証、外部通知、添付、承認、監査ログ、DB移行は見送る                               |
| ドキュメント重視   | 要件、データ、画面、業務フロー、アーキテクチャ、テスト、効果を説明できるようにする |

Ver.3開発は、以下のWBSで進める。

## 4. 推奨ブランチ運用

Ver.3では、以下のブランチ運用を推奨する。

| ブランチ                     | 用途                           |
| ---------------------------- | ------------------------------ |
| `main`                       | 安定版                         |
| `feature/ver3-wbs0-refactor` | WBS0〜WBS1の構造整理・設計整理 |
| `feature/ver3-enhancement`   | Ver.3本体実装                  |
| `feature/ver3-release`       | Ver.3リリース準備              |

WBS1完了後、Ver.3本体実装に進む場合は、以下のように新しい作業ブランチを作成する。

```bash
git switch feature/ver3-wbs0-refactor
git pull origin feature/ver3-wbs0-refactor
git switch -c feature/ver3-enhancement
git push -u origin feature/ver3-enhancement
```

| 通知機能が重くなる          | 外部送信や再送制御まで作りたくなる   | 通知対象抽出・通知文生成・ログ保存までに止める        |
| 履歴機能が監査ログ化する    | 完全な差分管理やアクセスログに広がる | コメント・ステータス履歴を中心にする                  |
| FAQ機能がナレッジ管理化する | 承認、版管理、推薦まで広がる         | 公開・検索・閲覧数・役立ち件数までに止める            |
| 集計が重くなる              | BI基盤や高度分析に寄る               | 簡易集計とCSV出力までにする                           |
| 設計文書が多すぎる          | 採用担当が読み切れない               | `ver3_portfolio_summary.md` とREADMEで要約する        |

## 19. 実装順序

実装順序は以下を推奨する。

```text
WBS0  構造整理
  ↓
WBS1  設計整理
  ↓
WBS2  DB拡張
  ↓
WBS3  簡易ログイン・ロール別表示
  ↓
WBS4  FAQ公開・検索
  ↓
WBS5  依頼者向け画面
  ↓
WBS6  コメント履歴・ステータス履歴・簡易操作履歴
  ↓
WBS7  通知対象抽出・通知文生成
  ↓
WBS8  Ver.3集計・Tableau出力拡張
  ↓
WBS9  テスト・ドキュメント更新
  ↓
WBS10 リリース準備・ポートフォリオ反映
```

通知対象抽出は、コメント履歴・ステータス履歴より後に実装する。理由は、通知や集計より先に、問い合わせ対応の経緯を残す仕組みを整える方が実務上の価値が高いためである。

## 23. 結論

Ver.3では、問い合わせ管理アプリを本格業務システムとして作り切るのではなく、ポートフォリオ用PoCとして、依頼者の自己解決、ロール別利用、対応経緯の可視化、対応漏れ候補の把握、改善分析を試作する。
実装の中心は、簡易ログイン、FAQ公開・検索、依頼者向け画面、コメント履歴、ステータス履歴である。操作履歴、通知対象抽出、通知文生成、通知ログ、Ver.3集計、Tableau出力拡張は、業務改善効果を説明するための簡易実装に留める。
本格認証、外部通知、添付ファイル、承認ワークフロー、完全監査ログ、PostgreSQL移行、Flask/Django/FastAPI化、AI分類はVer.3では対象外とする。これらは、本格導入時に既製品導入、外注開発、内製継続を比較する領域として扱う。
このWBSにより、Ver.3開発では「何でも作る」のではなく、「ポートフォリオとして評価されやすい範囲を、順序立てて実装する」ことを重視する。
