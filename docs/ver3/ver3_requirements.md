# Ver.3 要件定義書

## 1. 目的

本ドキュメントは、社内問い合わせ管理システム Ver.3 で追加・変更する要件を定義するものである。

Ver.1では、問い合わせの受付、記録、進捗管理、期限超過判定、基本集計、Tableau連携用CSV出力を実装した。
Ver.2では、Ver.1の機能を拡張し、要対応アラート、FAQ候補管理、カテゴリ別入力フォーム、依頼者向け確認画面、Ver.2追加集計、Tableau出力列の拡張を実装した。Ver.2.0.1では、Ver.2の完成状態を固定し、READMEおよび関連ドキュメントの整合性を修正した。

Ver.3では、Ver.2で蓄積できるようになった問い合わせ情報、FAQ候補、アラート情報、依頼者向け表示情報を、より実運用に近い形で活用できるようにする。

主な目的は以下である。

| 目的             | 内容                                                          |
| ---------------- | ------------------------------------------------------------- |
| 問い合わせ削減   | FAQ候補を依頼者向けに公開し、問い合わせ前の自己解決を支援する |
| 対応漏れ防止     | 期限前、期限超過、担当者未設定などを通知対象として抽出する    |
| 閲覧範囲の制御   | 簡易ログインとロール管理により、利用者ごとに表示範囲を分ける  |
| 更新経緯の可視化 | 操作履歴・ステータス履歴・コメント履歴を保存する              |
| 保守性向上       | WBS0で分割したUI構成を前提に、Ver.3機能を段階的に追加する     |
| 改善効果の可視化 | FAQ、通知、履歴に関する集計を追加し、Tableau出力に反映する    |

## 2. Ver.3開発の背景

Ver.2では、問い合わせ管理を単なる記録・進捗管理から、対応漏れ防止、FAQ候補蓄積、依頼者向け進捗確認、追加情報入力へ拡張した。

一方で、Ver.2の状態では以下の課題が残っている。

| 課題       | Ver.2の状態                                  | Ver.3での対応方針                                                |
| ---------- | -------------------------------------------- | ---------------------------------------------------------------- |
| FAQ活用    | FAQ候補を管理部側で蓄積する段階              | 依頼者が公開FAQを検索・閲覧できるようにする                      |
| 依頼者確認 | 問い合わせIDまたは依頼者名で検索する簡易画面 | 簡易ログイン後、自分の問い合わせだけ確認できるようにする         |
| アラート   | アプリ画面上で要対応案件を確認               | 通知対象として抽出し、通知文を生成できるようにする               |
| 権限管理   | 厳密なロール別表示制御は未実装               | 依頼者、管理部担当者、管理者、閲覧者で利用範囲を分ける           |
| 更新履歴   | 最新状態は確認できるが、変更経緯は追いにくい | 操作履歴・ステータス履歴・コメント履歴を追加する                 |
| データ構造 | inquiries テーブルにFAQ候補や追加情報も保持  | FAQ、ユーザー、履歴、通知を別テーブルとして整理する              |
| コード構造 | Ver.2までは `app.py` に画面処理が集中        | WBS0で `src/ui/` に画面を分割済み。Ver.3ではこの構成を前提にする |

## 3. Ver.3の位置づけ

Ver.3は、Ver.2の単なる機能追加ではなく、問い合わせ管理アプリを「管理部内の効率化ツール」から「依頼者・担当者・管理者が使う業務システム」に近づける段階である。

| バージョン | 位置づけ                                                             |
| ---------- | -------------------------------------------------------------------- |
| Ver.1      | 問い合わせを記録・管理する土台を作る                                 |
| Ver.2      | 対応漏れ防止、FAQ候補蓄積、依頼者確認、追加集計に拡張する            |
| Ver.3      | 蓄積した情報を、依頼者・担当者・管理者が活用できる運用機能へ接続する |

Ver.3の中心テーマは以下である。

```text
問い合わせ管理を、管理部内の効率化から、依頼者の自己解決と組織的な運用管理へ発展させる。
```

## 4. 開発範囲

Ver.3では、以下を開発対象とする。

| 優先度 | 機能                   | 内容                                                         |
| ------ | ---------------------- | ------------------------------------------------------------ |
| 高     | 簡易ログイン           | 利用者を識別し、ロールごとの画面表示に切り替える             |
| 高     | 簡易ロール管理         | 依頼者、管理部担当者、管理者、閲覧者で表示・操作範囲を分ける |
| 高     | FAQ公開・検索          | FAQ候補から公開FAQを作成し、依頼者が検索できるようにする     |
| 高     | 依頼者向け画面の本格化 | ログインユーザー本人の問い合わせだけ確認できるようにする     |
| 中     | 通知対象抽出           | 期限前、期限超過、担当者未設定、情報待ち長期化を抽出する     |
| 中     | 通知文生成             | 通知対象に応じた通知文を生成し、通知ログに保存する           |
| 中     | 操作履歴               | 問い合わせ作成・更新、FAQ公開変更などの操作履歴を保存する    |
| 中     | ステータス履歴         | 問い合わせごとのステータス変更履歴を保存する                 |
| 中     | コメント履歴           | 対応途中のメモや確認事項を問い合わせ単位で記録する           |
| 中     | Ver.3集計              | FAQ、通知、履歴に関する集計を追加する                        |
| 中     | Tableau出力拡張        | Ver.3追加項目をTableau用CSVに出力する                        |

## 5. 対象外

Ver.3では、以下は対象外とする。

| 対象外                      | 理由                                                           |
| --------------------------- | -------------------------------------------------------------- |
| 本格的なパスワード認証      | ポートフォリオ段階ではユーザー選択式の簡易ログインで十分なため |
| 外部メール・Slackへの実送信 | Ver.3では通知対象抽出と通知文生成までを対象とするため          |
| AIによる問い合わせ自動分類  | データ量が少ない段階では効果を説明しにくいため                 |
| 添付ファイル管理            | ファイル保存先、容量、権限管理の設計が別途必要なため           |
| PostgreSQL移行              | Ver.3ではSQLite継続とし、業務機能の拡張を優先するため          |
| Flask / Django / FastAPI化  | StreamlitのままでもVer.3の改善効果を示せるため                 |
| 本格承認ワークフロー        | 業務ルールが複雑化し、Ver.3の主目的から外れるため              |
| 多人数同時利用への厳密対応  | 小規模な業務改善デモとして扱うため                             |

## 6. 利用者・ロール

Ver.3では、以下の利用者を想定する。

| 利用者       | 役割                       | 主な利用目的                                            |
| ------------ | -------------------------- | ------------------------------------------------------- |
| 依頼者       | 問い合わせを行う社員       | FAQ検索、新規問い合わせ、自分の問い合わせ状況確認       |
| 管理部担当者 | 問い合わせ対応を行う担当者 | 担当案件確認、ステータス更新、コメント記録、FAQ候補登録 |
| 管理者       | 管理部長または運用管理者   | 全件確認、担当者設定、FAQ公開管理、履歴確認、集計確認   |
| 閲覧者       | 経営層・部門責任者         | 集計・Tableau出力結果の確認                             |

Ver.3では、以下のロールを用いる。

| ロール    | 想定利用者   | 主な権限                                                |
| --------- | ------------ | ------------------------------------------------------- |
| requester | 依頼者       | FAQ検索、新規問い合わせ、自分の問い合わせ確認           |
| staff     | 管理部担当者 | 担当案件確認、問い合わせ更新、コメント追加、FAQ候補登録 |
| admin     | 管理者       | 全件確認、担当者変更、FAQ公開管理、履歴確認、集計出力   |
| viewer    | 閲覧者       | 集計・CSV出力画面の閲覧                                 |

## 7. 機能要件

### 7.1 簡易ログイン機能

| ID         | 要件                                                         |
| ---------- | ------------------------------------------------------------ |
| FR-AUTH-01 | アプリ上で利用者を選択してログイン状態にできる               |
| FR-AUTH-02 | ログイン中のユーザー名、部署、ロールを画面上に表示する       |
| FR-AUTH-03 | `st.session_state` を用いてログイン状態を保持する            |
| FR-AUTH-04 | ユーザー切替またはログアウトができる                         |
| FR-AUTH-05 | ログインしていない状態では、管理系画面を表示しない           |
| FR-AUTH-06 | 本格的なパスワード認証は実装せず、デモ用の簡易ログインとする |

### 7.2 簡易ロール管理機能

| ID         | 要件                                                                            |
| ---------- | ------------------------------------------------------------------------------- |
| FR-ROLE-01 | ロールごとに利用可能な画面を切り替える                                          |
| FR-ROLE-02 | requester は公開FAQ、自分の問い合わせ、新規問い合わせ画面を利用できる           |
| FR-ROLE-03 | staff は問い合わせ一覧、ステータス更新、要対応アラート、FAQ候補管理を利用できる |
| FR-ROLE-04 | admin は全画面を利用できる                                                      |
| FR-ROLE-05 | viewer は集計・CSV出力画面のみ利用できる                                        |
| FR-ROLE-06 | 依頼者には管理部内部向け項目を表示しない                                        |
| FR-ROLE-07 | 依頼者は他者の問い合わせを閲覧できない                                          |
| FR-ROLE-08 | 権限外の操作は画面上で選択できないようにする                                    |

### 7.3 FAQ公開・検索機能

| ID        | 要件                                                               |
| --------- | ------------------------------------------------------------------ |
| FR-FAQ-01 | 管理者はFAQ候補から公開FAQを作成できる                             |
| FR-FAQ-02 | 管理者はFAQの公開・非公開を切り替えられる                          |
| FR-FAQ-03 | 管理者はFAQタイトル、カテゴリ、回答内容を編集できる                |
| FR-FAQ-04 | 依頼者は公開FAQを一覧表示できる                                    |
| FR-FAQ-05 | 依頼者はFAQをキーワードで検索できる                                |
| FR-FAQ-06 | 依頼者はFAQをカテゴリで絞り込める                                  |
| FR-FAQ-07 | FAQ詳細には、FAQタイトル、カテゴリ、回答内容、最終更新日を表示する |
| FR-FAQ-08 | FAQ閲覧時に閲覧回数を記録できる                                    |
| FR-FAQ-09 | FAQが役に立った場合、役立ち件数を記録できる                        |
| FR-FAQ-10 | FAQで解決しない場合、新規問い合わせ画面へ進める                    |

### 7.4 依頼者向け画面本格化

| ID        | 要件                                                                                                                 |
| --------- | -------------------------------------------------------------------------------------------------------------------- |
| FR-REQ-01 | 依頼者用トップ画面を用意する                                                                                         |
| FR-REQ-02 | 依頼者トップからFAQ検索、新規問い合わせ、自分の問い合わせ一覧へ移動できる                                            |
| FR-REQ-03 | 依頼者はログインユーザー本人の問い合わせだけ確認できる                                                               |
| FR-REQ-04 | 依頼者向け一覧には、問い合わせID、受付日、カテゴリ、ステータス、回答概要、完了日を表示する                           |
| FR-REQ-05 | 管理部内部メモ、操作履歴、担当者用コメントは依頼者画面には表示しない                                                 |
| FR-REQ-06 | Ver.2の依頼者名検索方式は、本格運用前の簡易確認機能として位置づけ、Ver.3ではログインユーザーに紐づく表示に置き換える |

### 7.5 通知対象抽出・通知文生成機能

Ver.3では、メールやSlackへの実送信は対象外とし、通知対象の抽出、通知文生成、通知ログ保存までを対象とする。

| ID           | 要件                                                                     |
| ------------ | ------------------------------------------------------------------------ |
| FR-NOTIFY-01 | 期限前通知の対象問い合わせを抽出できる                                   |
| FR-NOTIFY-02 | 期限超過通知の対象問い合わせを抽出できる                                 |
| FR-NOTIFY-03 | 担当者未設定通知の対象問い合わせを抽出できる                             |
| FR-NOTIFY-04 | 情報待ち長期化通知の対象問い合わせを抽出できる                           |
| FR-NOTIFY-05 | ステータス変更時に依頼者向け通知文を生成できる                           |
| FR-NOTIFY-06 | 通知対象、通知種別、通知文面、通知対象者、作成日時を通知ログに保存できる |

## 9. データ要件

Ver.3では、既存の `inquiries` テーブルを問い合わせ本体として継続利用する。
そのうえで、以下のテーブルを追加する。

| テーブル            | 内容                               |
| ------------------- | ---------------------------------- |
| `users`             | 利用者情報とロールを管理する       |
| `faq_items`         | 公開FAQを管理する                  |
| `inquiry_comments`  | 問い合わせごとのコメントを管理する |
| `status_history`    | ステータス変更履歴を管理する       |
| `operation_logs`    | 操作履歴を管理する                 |
| `notification_logs` | 通知対象・通知文生成履歴を管理する |

Ver.3では、`roles` テーブルは作成しない。ロールは `users.role` に requester、staff、admin、viewer のいずれかを保存する簡易方式とする。

## 10. 非機能要件

### 10.1 保守性

| ID           | 要件                                                                                        |
| ------------ | ------------------------------------------------------------------------------------------- |
| NFR-MAINT-01 | `app.py` はアプリの入口、データ読み込み、タブ構成、各画面呼び出しを担当する                 |
| NFR-MAINT-02 | 画面表示処理は `src/ui/` 配下に配置する                                                     |
| NFR-MAINT-03 | 業務ロジックは `src/services/` 配下へ段階的に分離する                                       |
| NFR-MAINT-04 | DB操作は当面 `src/db.py` を継続利用し、必要に応じて後続バージョンでrepository分離を検討する |
| NFR-MAINT-05 | 既存Ver.2.0.1機能を壊さずに拡張する                                                         |
| NFR-MAINT-06 | 主要な設計判断はドキュメントに残す                                                          |

### 10.2 テスト容易性

| ID          | 要件                                                                |
| ----------- | ------------------------------------------------------------------- |
| NFR-TEST-01 | Ver.2.0.1の既存スモークテストが継続して通る                         |
| NFR-TEST-02 | Ver.3用スモークテストを追加する                                     |
| NFR-TEST-03 | DB初期化後にVer.3追加テーブルが作成されることを確認する             |
| NFR-TEST-04 | FAQ公開、簡易ログイン、通知対象抽出、履歴保存の基本動作を確認できる |
| NFR-TEST-05 | Tableau用CSV出力にVer.3追加列が含まれることを確認する               |
| NFR-TEST-06 | 画面操作用の手動テストチェックリストを作成する                      |

### 10.3 セキュリティ・権限

| ID         | 要件                                                                      |

## 16. 結論

Ver.3では、簡易ログイン、簡易ロール管理、FAQ公開・検索、依頼者向け画面の本格化、通知対象抽出、通知文生成、操作履歴、ステータス履歴、コメント履歴、Ver.3集計、Tableau出力拡張を中心に実装する。

Ver.3の主題は、新機能を単に増やすことではない。Ver.2で蓄積できるようになった問い合わせ情報・FAQ候補・アラート情報を、依頼者、管理部担当者、管理者がそれぞれ活用できる運用機能へ接続することである。

これにより、本システムは「問い合わせを記録・管理するアプリ」から、「問い合わせを減らし、対応漏れを防ぎ、更新経緯を追跡し、組織的に改善を回す業務システム」へ発展する。
