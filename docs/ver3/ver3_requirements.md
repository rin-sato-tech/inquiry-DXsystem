# Ver.3 要件定義書

## 1. 目的

本ドキュメントは、社内問い合わせ管理システム Ver.3 で追加・変更する要件を定義するものである。

Ver.3は、本格的な問い合わせ管理SaaSや社内業務システムを置き換えることを目的としない。本システムは、メール・Excel・口頭・チャットで分散していた社内問い合わせ対応を題材に、業務フロー整理、データ構造化、対応漏れ防止、FAQ化、履歴管理、集計・可視化までを試作するポートフォリオ用PoCである。
そのため、Ver.3ではすべての機能を本格実装するのではなく、以下を明確に分ける。

| 区分           | 方針                                                   |
| -------------- | ------------------------------------------------------ |
| 必須実装       | ポートフォリオ価値が高く、業務改善の説明に直結するもの |
| 簡易実装       | 実務感は出るが、作り込みすぎると過剰になるもの         |
| 設計・将来課題 | 文書上は整理するが、Ver.3では深追いしないもの          |
| 対象外         | Ver.3では作らないもの                                  |

Ver.3の目的は、完成された本番システムを作ることではなく、**現場課題を分析し、小さく試作し、どこまで内製し、どこから既製品導入・外注を検討すべきか判断できる状態を示すこと**である。

---

## 2. Ver.3の位置づけ

Ver.1では、問い合わせの受付、記録、進捗管理、期限超過判定、基本集計、Tableau連携用CSV出力を実装した。
Ver.2では、Ver.1の機能を拡張し、要対応アラート、FAQ候補管理、カテゴリ別入力フォーム、依頼者向け確認画面、Ver.2追加集計、Tableau出力列の拡張を実装した。

Ver.3では、Ver.2で蓄積できるようになった問い合わせ情報、FAQ候補、アラート情報、依頼者向け表示情報を、より実運用に近い形で活用できるようにする。
ただし、Ver.3は本格運用システムではなく、**本格導入前のPoC拡張**として位置づける。

| バージョン | 位置づけ                                                           |
| ---------- | ------------------------------------------------------------------ |
| Ver.1      | 問い合わせを記録・管理する土台を作る                               |
| Ver.2      | 対応漏れ防止、FAQ候補蓄積、依頼者確認、追加集計に拡張する          |
| Ver.3      | 依頼者の自己解決、簡易ロール管理、履歴管理、通知対象抽出を試作する |
| 本格導入   | 既製品導入、外注開発、内製継続を比較検討する                       |

Ver.3の中心テーマは以下である。

> 問い合わせ管理を、管理部内の効率化から、依頼者の自己解決と組織的な改善判断へ発展させる。

## 3. Ver.3開発の背景

Ver.2では、問い合わせ管理を単なる記録・進捗管理から、対応漏れ防止、FAQ候補蓄積、依頼者向け進捗確認、カテゴリ別追加情報入力へ拡張した。

一方で、Ver.2の状態では以下の課題が残っている。

| 課題       | Ver.2の状態                                   | Ver.3での対応方針                                                |
| ---------- | --------------------------------------------- | ---------------------------------------------------------------- |
| FAQ活用    | FAQ候補を管理部側で蓄積する段階               | 公開FAQとして依頼者が検索できるようにする                        |
| 依頼者確認 | 問い合わせIDまたは依頼者名で検索する簡易画面  | 簡易ログイン後、自分の問い合わせだけ確認できるようにする         |
| アラート   | アプリ画面上で要対応案件を確認                | 通知対象として抽出し、通知文を生成できるようにする               |
| 権限管理   | 厳密なロール別表示制御は未実装                | 依頼者、管理部担当者、管理者、閲覧者で画面表示を分ける           |
| 更新履歴   | 最新状態は確認できるが、変更経緯は追いにくい  | ステータス履歴・コメント履歴を追加する                           |
| データ構造 | `inquiries` テーブルにFAQ候補や追加情報も保持 | FAQ、ユーザー、履歴、通知を別テーブルとして整理する              |
| コード構造 | Ver.2までは `app.py` に画面処理が集中         | WBS0で `src/ui/` に画面を分割済み。Ver.3ではこの構成を前提にする |

## 4. Ver.3実装範囲の分類

Ver.3では、設計上の項目をすべて同じ深さで作り込むのではなく、以下の分類で実装する。

### 4.1 必須実装

以下は、Ver.3で優先的に実装する。

| 機能                 | 理由                                                            | 実装ライン                                                              |
| -------------------- | --------------------------------------------------------------- | ----------------------------------------------------------------------- |
| DB拡張               | Ver.3機能の土台になるため                                       | `users`, `faq_items`, `inquiry_comments`, `status_history` を中心に追加 |
| 簡易ログイン         | 利用者視点を分けるため                                          | ユーザー選択式ログイン                                                  |
| 簡易ロール管理       | 立場ごとに見える画面を変えるため                                | 画面表示制御まで                                                        |
| FAQ公開・検索        | 問い合わせ削減の効果を示しやすいため                            | 公開FAQ一覧、検索、詳細表示                                             |
| 依頼者トップ         | FAQ検索・問い合わせ登録・自分の問い合わせ確認への入口にするため | シンプルな導線画面                                                      |
| 自分の問い合わせ一覧 | 進捗確認の往復削減を説明できるため                              | ログインユーザー本人分だけ表示                                          |
| コメント履歴         | 引き継ぎ・対応経緯を残せるため                                  | 問い合わせ単位の簡易コメント                                            |
| ステータス履歴       | 状態変更の経緯を追えるため                                      | 変更前後のステータスを保存                                              |
| Ver.3テスト          | 品質確認の姿勢を示すため                                        | スモークテスト・手動テスト                                              |

---

### 4.2 簡易実装

以下は、Ver.3で実装するが、作り込みすぎない。

| 機能            | 理由                             | 実装ライン                                           |
| --------------- | -------------------------------- | ---------------------------------------------------- |
| 操作履歴        | 主要操作の追跡性を示すため       | 問い合わせ作成・更新、FAQ公開変更など主要操作のみ    |
| 通知対象抽出    | 対応漏れ防止を説明できるため     | 期限前、期限超過、担当者未設定、情報待ち長期化の抽出 |
| 通知文生成      | 外部通知前の業務設計を示せるため | 画面表示とログ保存まで                               |
| 通知ログ        | 通知対象の記録を残すため         | `created`, `reviewed`, `skipped` 程度の状態管理      |
| Ver.3集計       | 改善効果を数値で見せるため       | FAQ件数、通知件数、履歴件数の簡易集計                |
| Tableau出力拡張 | BI連携の継続性を示すため         | Ver.3指標を数列追加                                  |

操作履歴と通知ログは、本格的な監査・通知基盤として作るものではない。あくまで、問い合わせ対応の経緯や通知対象を確認できる程度の簡易実装とする。

---

### 4.3 設計・将来課題に留めるもの

以下は、Ver.3では設計方針を文書化するが、実装は必須としない。

| 項目                       | Ver.3での扱い                                              |
| -------------------------- | ---------------------------------------------------------- |
| `roles` テーブル           | 作らない。`users.role` で管理する                          |
| `user_roles` テーブル      | 作らない。1ユーザー1ロールで十分とする                     |
| 担当者ID化                 | `inquiries.assignee` は当面文字列で維持する                |
| `additional_info` の正規化 | TEXT形式を継続し、JSON化・子テーブル化は将来課題           |
| FAQ閲覧詳細ログ            | 作らない。`view_count`, `helpful_count` で十分とする       |
| 通知送信キュー             | 作らない。外部送信を行わないため不要                       |
| repository分離             | 設計上は検討するが、Ver.3必須ではない                      |
| Docker化                   | 必須ではない。必要に応じて将来課題とする                   |
| クラウドデプロイ           | 必須ではない。ポートフォリオではREADMEと画面説明を優先する |

---

### 4.4 対象外

以下は、Ver.3では作らない。

| 対象外                     | 理由                                                     |
| -------------------------- | -------------------------------------------------------- |
| 本格的なパスワード認証     | セキュリティ設計・アカウント管理が主題になってしまうため |
| OAuth・社内認証連携        | ポートフォリオ用PoCの範囲を超えるため                    |
| 外部メール送信             | 外部連携・送信設定・運用管理が必要になるため             |
| Slack / Teams送信          | 外部通知基盤の設計が必要になるため                       |
| AIによる問い合わせ自動分類 | データ量・精度検証・モデル評価が必要になるため           |
| 添付ファイル管理           | 保存先、容量、権限、ウイルス対策などが必要になるため     |
| PostgreSQL移行             | DB運用・インフラ寄りになり、Ver.3の主目的から外れるため  |
| Flask / Django / FastAPI化 | StreamlitのままでもPoC目的は達成できるため               |
| 本格承認ワークフロー       | 問い合わせ管理からワークフロー製品に近づきすぎるため     |
| 多人数同時利用への厳密対応 | 本番Webアプリ設計の領域になるため                        |
| 完全な監査ログ             | 内部統制システムの領域になるため                         |

## 5. 利用者・ロール

Ver.3では、以下の利用者を想定する。

| 利用者       | 役割                       | 主な利用目的                                            |
| ------------ | -------------------------- | ------------------------------------------------------- |
| 依頼者       | 問い合わせを行う社員       | FAQ検索、新規問い合わせ、自分の問い合わせ状況確認       |
| 管理部担当者 | 問い合わせ対応を行う担当者 | 担当案件確認、ステータス更新、コメント記録、FAQ候補登録 |
| 管理者       | 管理部長または運用管理者   | 全件確認、FAQ公開管理、履歴確認、集計確認               |
| 閲覧者       | 経営層・部門責任者         | 集計・Tableau出力結果の確認                             |

Ver.3では、以下のロールを用いる。

| ロール      | 想定利用者   | 主な権限                                                |
| ----------- | ------------ | ------------------------------------------------------- |
| `requester` | 依頼者       | FAQ検索、新規問い合わせ、自分の問い合わせ確認           |
| `staff`     | 管理部担当者 | 担当案件確認、問い合わせ更新、コメント追加、FAQ候補登録 |
| `admin`     | 管理者       | 全件確認、担当者変更、FAQ公開管理、履歴確認、集計出力   |
| `viewer`    | 閲覧者       | 集計・CSV出力画面の閲覧                                 |

ロール管理は簡易方式とし、`users.role` に `requester`, `staff`, `admin`, `viewer` のいずれかを保存する。`roles` テーブルや `user_roles` テーブルは作成しない。

## 6. 機能要件

### 6.1 簡易ログイン機能

| ID         | 実装区分     | 要件                                                   |
| ---------- | ------------ | ------------------------------------------------------ |
| FR-AUTH-01 | 必須         | アプリ上で利用者を選択してログイン状態にできる         |
| FR-AUTH-02 | 必須         | ログイン中のユーザー名、部署、ロールを画面上に表示する |
| FR-AUTH-03 | 必須         | `st.session_state` を用いてログイン状態を保持する      |
| FR-AUTH-04 | 必須         | ユーザー切替またはログアウトができる                   |
| FR-AUTH-05 | 必須         | ログインしていない状態では、管理系画面を表示しない     |
| FR-AUTH-06 | 対象外の明示 | 本格的なパスワード認証は実装しない                     |

### 6.2 簡易ロール管理機能

| ID         | 実装区分     | 要件                                                                              |
| ---------- | ------------ | --------------------------------------------------------------------------------- |
| FR-ROLE-01 | 必須         | ロールごとに利用可能な画面を切り替える                                            |
| FR-ROLE-02 | 必須         | `requester` は公開FAQ、自分の問い合わせ、新規問い合わせ画面を利用できる           |
| FR-ROLE-03 | 必須         | `staff` は問い合わせ一覧、ステータス更新、要対応アラート、FAQ候補管理を利用できる |
| FR-ROLE-04 | 必須         | `admin` は全画面を利用できる                                                      |
| FR-ROLE-05 | 必須         | `viewer` は集計・CSV出力画面のみ利用できる                                        |
| FR-ROLE-06 | 必須         | 依頼者には管理部内部向け項目を表示しない                                          |
| FR-ROLE-07 | 必須         | 依頼者は他者の問い合わせを閲覧できない                                            |
| FR-ROLE-08 | 簡易         | 権限外の操作は画面上で選択できないようにする                                      |
| FR-ROLE-09 | 対象外の明示 | 詳細な権限マスタ管理は実装しない                                                  |

### 6.4 依頼者向け画面

### 6.5 コメント履歴機能

| ID            | 実装区分 | 要件                                                 |
| ------------- | -------- | ---------------------------------------------------- |
| FR-COMMENT-01 | 必須     | 管理部担当者は問い合わせにコメントを追加できる       |
| FR-COMMENT-02 | 必須     | コメントには投稿者、投稿日時、本文を保存する         |
| FR-COMMENT-03 | 必須     | コメントは問い合わせ詳細画面で時系列に表示する       |
| FR-COMMENT-04 | 必須     | コメントには内部向け、依頼者向けの表示区分を持たせる |
| FR-COMMENT-05 | 必須     | 内部向けコメントは依頼者画面には表示しない           |
| FR-COMMENT-06 | 簡易     | 管理者は全コメントを確認できる                       |

コメント履歴は、チャット機能として作り込むものではない。対応経緯を残すための簡易メモ機能として実装する。

| FR-STATUS-03 | 必須         | 問い合わせ詳細画面でステータス履歴を確認できる                   |
| FR-STATUS-04 | 必須         | ステータス履歴は管理部担当者と管理者が閲覧できる                 |
| FR-STATUS-05 | 必須         | 依頼者には内部向けのステータス履歴を表示しない                   |

### 6.7 操作履歴機能

| ID        | 実装区分     | 要件                                                                       |
| --------- | ------------ | -------------------------------------------------------------------------- |
| FR-LOG-01 | 簡易         | 問い合わせ作成時に操作履歴を保存する                                       |
| FR-LOG-02 | 簡易         | 問い合わせ更新時に操作履歴を保存する                                       |
| FR-LOG-03 | 簡易         | FAQ作成・更新・公開変更時に操作履歴を保存する                              |
| FR-LOG-04 | 簡易         | 操作履歴には、操作日時、操作ユーザー、操作種別、対象ID、操作内容を保存する |
| FR-LOG-05 | 簡易         | 管理者は操作履歴一覧を確認できる                                           |
| FR-LOG-06 | 対象外の明示 | 内部統制レベルの完全な監査ログは実装しない                                 |

操作履歴は、主要操作を追跡するための簡易ログとして扱う。本格的な監査証跡、改ざん防止、アクセスログ管理は対象外とする。

### 6.8 通知対象抽出・通知文生成機能

Ver.3では、メールやSlackへの実送信は対象外とし、通知対象の抽出、通知文生成、通知ログ保存までを対象とする。

| ID           | 実装区分     | 要件                                                                     |
| ------------ | ------------ | ------------------------------------------------------------------------ |
| FR-NOTIFY-01 | 簡易         | 期限前通知の対象問い合わせを抽出できる                                   |
| FR-NOTIFY-02 | 簡易         | 期限超過通知の対象問い合わせを抽出できる                                 |
| FR-NOTIFY-03 | 簡易         | 担当者未設定通知の対象問い合わせを抽出できる                             |
| FR-NOTIFY-04 | 簡易         | 情報待ち長期化通知の対象問い合わせを抽出できる                           |
| FR-NOTIFY-05 | 簡易         | ステータス変更時に依頼者向け通知文を生成できる                           |
| FR-NOTIFY-06 | 簡易         | 通知対象、通知種別、通知文面、通知対象者、作成日時を通知ログに保存できる |
| FR-NOTIFY-07 | 簡易         | 管理部担当者または管理者は通知対象一覧を確認できる                       |
| FR-NOTIFY-08 | 将来課題     | 外部通知連携を想定し、通知文生成処理を関数化する                         |
| FR-NOTIFY-09 | 対象外の明示 | メール、Slack、Teamsへの実送信は行わない                                 |
| FR-NOTIFY-10 | 対象外の明示 | 通知キュー、再送制御、送信失敗管理は実装しない                           |

### 6.9 Ver.3集計機能

| ID        | 実装区分     | 要件                                                     |
| --------- | ------------ | -------------------------------------------------------- |
| FR-SUM-01 | 簡易         | 公開FAQ件数を集計できる                                  |
| FR-SUM-02 | 簡易         | カテゴリ別公開FAQ件数を集計できる                        |
| FR-SUM-03 | 簡易         | FAQ閲覧数を集計できる                                    |
| FR-SUM-04 | 簡易         | FAQ役立ち件数を集計できる                                |
| FR-SUM-05 | 簡易         | 通知種別別の通知対象件数を集計できる                     |
| FR-SUM-06 | 簡易         | ステータス変更件数を集計できる                           |

## 14. 実運用時の導入判断

Ver.3はPoCであり、本格運用に進む場合は以下の選択肢を比較する必要がある。

| 選択肢       | 向いている状況                                                 |
| ------------ | -------------------------------------------------------------- |
| 内製継続     | 小規模部署内で利用し、要件が限定的な場合                       |
| 既製品導入   | 複数部署・全社で利用し、認証・権限・通知・添付管理が必要な場合 |
| 外注開発     | 自社固有の業務フローが強く、既製品では対応しにくい場合         |
| 現行運用改善 | システム化より、FAQ整備や業務ルール変更の方が効果的な場合      |

Ver.3では、上記判断の前段階として、現場課題を可視化し、必要機能を整理することを目的とする。

## 16. 結論

Ver.3では、簡易ログイン、簡易ロール管理、FAQ公開・検索、依頼者向け画面、コメント履歴、ステータス履歴を中心に実装する。
通知対象抽出、通知文生成、操作履歴、通知ログ、Ver.3集計、Tableau出力拡張は、業務改善効果を説明するための補助機能として簡易実装する。
一方で、本格認証、外部通知、添付ファイル、承認ワークフロー、PostgreSQL移行、AI分類、本格的な監査ログはVer.3では作らない。これらは、本格導入時に既製品導入・外注開発・内製継続を比較する領域として扱う。
Ver.3の主題は、新機能を大量に増やすことではない。Ver.2で蓄積できるようになった問い合わせ情報・FAQ候補・アラート情報を、依頼者、管理部担当者、管理者が活用できるPoCとして整理し、業務改善の方向性と導入判断を説明できる状態にすることである。
