# Ver.3 業務フロー

## 1. 目的

本ドキュメントは、社内問い合わせ管理システム Ver.3 における業務フローを整理するものである。

Ver.3では、Ver.2.0.1までの問い合わせ登録、進捗管理、要対応アラート、FAQ候補管理、依頼者向け確認、集計・CSV出力を前提とし、以下の業務フローを追加・拡張する。

| フロー             | 内容                                                          |
| ------------------ | ------------------------------------------------------------- |
| 依頼者フロー       | FAQ検索、自己解決、新規問い合わせ、自分の問い合わせ確認       |
| 管理部担当者フロー | 要対応アラート確認、問い合わせ更新、コメント記録、FAQ候補作成 |
| 管理者フロー       | FAQ公開管理、履歴確認、通知対象確認、集計確認                 |
| 通知対象確認フロー | 期限前、期限超過、担当者未設定などの通知対象を確認する        |
| 履歴確認フロー     | 操作履歴、ステータス履歴、コメント履歴を確認する              |
| 集計・改善フロー   | FAQ活用状況、通知対象件数、履歴情報を分析する                 |

---

## 2. Ver.2までの業務フロー

Ver.2までの主な業務フローは以下である。

### 2.1 問い合わせ登録フロー

1. 依頼者または管理部担当者が問い合わせ内容を登録する
2. カテゴリ、優先度、希望期限、担当者などを入力する
3. カテゴリ別追加項目を入力する
4. 問い合わせが `inquiries` に保存される
5. 管理部担当者が一覧画面で確認する

### 2.2 対応管理フロー

1. 管理部担当者が問い合わせ一覧を確認する
2. 担当者、ステータス、対応概要を更新する
3. 必要に応じて完了日を入力する
4. 集計画面で件数、期限超過、対応時間を確認する

### 2.3 Ver.2追加フロー

| フロー         | 内容                                                     |
| -------------- | -------------------------------------------------------- |
| 要対応アラート | 期限超過、本日期限、期限間近、担当者未設定などを確認する |
| FAQ候補管理    | 完了済み問い合わせからFAQ候補を登録する                  |
| 依頼者向け確認 | 問い合わせIDまたは依頼者名で状況を確認する               |
| Ver.2集計      | FAQ候補、追加情報入力状況、依頼者向け表示対象を集計する  |

---

## 3. Ver.3で変わる業務フロー

Ver.3では、Ver.2の管理部中心のフローを、依頼者・管理部担当者・管理者がそれぞれ利用するフローへ拡張する。

| 観点         | Ver.2まで                    | Ver.3                                         |
| ------------ | ---------------------------- | --------------------------------------------- |
| 問い合わせ前 | 依頼者は問い合わせを登録する | 依頼者はまずFAQを検索する                     |
| FAQ          | 管理部側でFAQ候補を蓄積      | 管理者が公開FAQとして整備し、依頼者が検索する |
| 依頼者確認   | IDまたは依頼者名で検索       | ログイン後、自分の問い合わせだけ確認する      |
| アラート     | 管理部が画面で確認           | 通知対象として抽出し、通知文を生成する        |
| 更新履歴     | 最新状態のみ確認             | 操作履歴・ステータス履歴を保存する            |
| 対応メモ     | 対応概要に集約               | コメント履歴として時系列で保存する            |
| 管理分析     | 件数・対応時間中心           | FAQ活用、通知対象、履歴情報も分析する         |

---

## 4. ロール別の基本フロー

Ver.3では、以下のロールごとに業務フローを分ける。

| ロール    | 主なフロー                                                                  |
| --------- | --------------------------------------------------------------------------- |
| requester | FAQ検索 → 自己解決または問い合わせ登録 → 自分の問い合わせ確認               |
| staff     | アラート確認 → 問い合わせ対応 → コメント記録 → ステータス更新 → FAQ候補登録 |
| admin     | 全件確認 → FAQ公開管理 → 通知対象確認 → 履歴確認 → 集計確認                 |
| viewer    | 集計・CSV出力の確認                                                         |

## 5. 依頼者フロー

### 5.1 目的

依頼者が問い合わせ前にFAQを確認し、自己解決できる場合は問い合わせを行わずに済むようにする。問い合わせが必要な場合は、登録後に自分の問い合わせ状況を確認できるようにする。

### 5.2 基本フロー

```text
依頼者ログイン
  ↓
依頼者トップ表示
  ↓
FAQ検索
  ↓
解決できたか？
  ├─ はい → 問い合わせせず終了
  └─ いいえ → 新規問い合わせ登録
                  ↓
              自分の問い合わせ一覧で状況確認
                  ↓
              回答・完了を確認
```

### 5.3 詳細手順

| Step | 実施者 | 操作                                        | システム処理                                                    |
| ---: | ------ | ------------------------------------------- | --------------------------------------------------------------- |
|    1 | 依頼者 | ユーザーを選択してログインする              | `st.session_state` にユーザー情報を保存する                     |
|    2 | 依頼者 | 依頼者トップを開く                          | FAQ検索・新規問い合わせ・自分の問い合わせ確認への導線を表示する |
|    3 | 依頼者 | FAQ検索画面でキーワード・カテゴリを入力する | 公開FAQを検索する                                               |
|    4 | 依頼者 | FAQ詳細を確認する                           | FAQ閲覧数を加算する                                             |
|    5 | 依頼者 | FAQが役に立った場合、役立ちボタンを押す     | 役立ち件数を加算する                                            |
|    6 | 依頼者 | 解決しない場合、新規問い合わせへ進む        | 新規問い合わせ画面を表示する                                    |
|    7 | 依頼者 | 問い合わせ内容を登録する                    | `inquiries` に登録し、操作履歴を保存する                        |
|    8 | 依頼者 | 自分の問い合わせ一覧を確認する              | ログインユーザー本人の問い合わせだけ表示する                    |
|    9 | 依頼者 | ステータス・回答概要を確認する              | 依頼者向け表示項目のみ表示する                                  |

### 5.4 依頼者に表示する情報

| 表示項目       | 内容                   |
| -------------- | ---------------------- |
| 問い合わせID   | 問い合わせを識別するID |
| 受付日         | 問い合わせ登録日       |
| カテゴリ       | 問い合わせカテゴリ     |
| 問い合わせ概要 | 依頼内容の要約         |
| ステータス     | 現在の対応状況         |
| 回答概要       | 管理部からの回答       |
| 完了日         | 完了した日             |

### 5.5 依頼者に表示しない情報

| 非表示情報               | 理由                           |
| ------------------------ | ------------------------------ |
| 内部コメント             | 管理部内部の対応メモであるため |
| 操作履歴                 | 内部管理用情報であるため       |
| ステータス変更履歴の詳細 | 管理部の対応経緯であるため     |
| 担当者の内部作業メモ     | 依頼者向けではないため         |
| 非公開FAQ                | 管理者確認前の情報であるため   |

## 6. 管理部担当者フロー

### 6.1 目的

管理部担当者が、対応すべき問い合わせを把握し、対応状況を更新し、必要なコメントやFAQ候補を残せるようにする。

### 6.2 基本フロー

```text
管理部担当者ログイン
  ↓
要対応アラート確認
担当案件または未割当案件を確認
  ↓
問い合わせ更新
コメント追加・ステータス更新
  ↓
完了後、FAQ候補にできるか判断
必要に応じてFAQ候補登録
```

### 6.3 詳細手順

| Step | 実施者       | 操作                           | システム処理                                             |
| ---: | ------------ | ------------------------------ | -------------------------------------------------------- |
|    1 | 管理部担当者 | ユーザーを選択してログインする | staffロールとしてメニューを表示する                      |
|    2 | 管理部担当者 | 要対応アラート画面を開く       | 期限超過、本日期限、期限間近、担当者未設定などを表示する |
|    3 | 管理部担当者 | 対応対象を選択する             | 問い合わせ更新画面へ進む                                 |
|    4 | 管理部担当者 | 担当者を設定する               | `inquiries.assignee` を更新し、操作履歴を保存する        |
|    5 | 管理部担当者 | ステータスを更新する           | `inquiries.status` を更新し、`status_history` に保存する |
|    6 | 管理部担当者 | コメントを追加する             | `inquiry_comments` に保存する                            |
|    7 | 管理部担当者 | 対応概要を入力する             | `response_summary` を更新する                            |
|    8 | 管理部担当者 | 完了処理を行う                 | 完了日を保存し、必要に応じてステータス履歴を保存する     |
|    9 | 管理部担当者 | FAQ候補として残すか判断する    | FAQ候補フラグ、FAQタイトル、回答案を登録する             |

### 6.4 管理部担当者の判断ポイント

| 判断ポイント | 内容                                                         |
| ------------ | ------------------------------------------------------------ |
| 優先度       | 期限、影響範囲、依頼部署を見て判断する                       |
| 担当者設定   | 未割当案件を放置しない                                       |
| 情報待ち     | 依頼者からの追加情報が必要な場合、ステータスを情報待ちにする |
| FAQ候補      | 同種問い合わせが再発しそうなものをFAQ候補にする              |
| コメント区分 | 内部向けコメントと依頼者向けコメントを分ける                 |

## 7. 管理者フロー

### 7.1 目的

管理者が、問い合わせ全体の状況、FAQ公開、通知対象、履歴、集計を確認し、組織的な改善につなげる。

### 7.2 基本フロー

```text
管理者ログイン
  ↓
問い合わせ全体・アラート確認
FAQ候補を確認
公開FAQとして整備
  ↓
通知対象・履歴を確認
集計・CSV出力で改善状況を確認
```

### 7.3 詳細手順

| Step | 実施者 | 操作                               | システム処理                                 |
| ---: | ------ | ---------------------------------- | -------------------------------------------- |
|    1 | 管理者 | adminユーザーでログインする        | 全画面を利用可能にする                       |
|    2 | 管理者 | 問い合わせ一覧を確認する           | 全問い合わせを表示する                       |
|    3 | 管理者 | 要対応アラートを確認する           | 期限超過、未割当、情報待ち長期化を確認する   |
|    4 | 管理者 | FAQ候補を確認する                  | FAQ候補一覧を表示する                        |
|    5 | 管理者 | FAQタイトル・回答を編集する        | `faq_items` を作成または更新する             |
|    6 | 管理者 | FAQを公開する                      | `is_public = 1` に更新し、操作履歴を保存する |
|    7 | 管理者 | 通知対象を確認する                 | 通知対象抽出・通知文生成を行う               |
|    8 | 管理者 | 操作履歴・ステータス履歴を確認する | 履歴確認画面で検索・絞り込みを行う           |
|    9 | 管理者 | 集計画面を確認する                 | FAQ、通知、履歴に関する集計を確認する        |
|   10 | 管理者 | Tableau用CSVを出力する             | Ver.3追加項目を含むCSVを生成する             |

### 7.4 管理者の判断ポイント

| 判断ポイント | 内容                                                           |
| ------------ | -------------------------------------------------------------- |
| FAQ公開可否  | 依頼者向けに公開して問題ない内容か確認する                     |
| 個人情報除去 | FAQ文面に個人名・部署固有情報が残っていないか確認する          |
| アラート傾向 | 期限超過や未割当が特定担当者・カテゴリに偏っていないか確認する |
| 通知対象     | 通知が必要な案件か、除外すべき案件か確認する                   |
| 業務改善     | FAQ化、マニュアル整備、担当者配分の見直しにつなげる            |

## 8. 閲覧者フロー

### 8.1 目的

経営層や部門責任者が、問い合わせ対応状況や改善効果を把握できるようにする。

### 8.2 基本フロー

```text
閲覧者ログイン
  ↓
集計・CSV出力画面を確認
  ↓
問い合わせ傾向・FAQ活用状況・通知対象件数を確認
  ↓
必要に応じてTableauで可視化
```

### 8.3 表示内容

| 表示情報       | 内容                               |
| -------------- | ---------------------------------- |
| 問い合わせ件数 | 全体件数、カテゴリ別件数           |
| 対応状況       | 未完了、完了、期限超過             |
| FAQ活用状況    | 公開FAQ件数、閲覧数、役立ち件数    |
| 通知対象件数   | 期限前、期限超過、担当者未設定など |
| 履歴集計       | 操作件数、ステータス変更件数       |
| Tableau用CSV   | 可視化用データ出力                 |

閲覧者は問い合わせの更新、FAQ編集、履歴詳細確認は行わない。

## 9. FAQ公開フロー

### 9.1 目的

Ver.2で蓄積したFAQ候補を、依頼者が利用できる公開FAQへ変換する。

### 9.2 基本フロー

```text
問い合わせ完了
  ↓
管理部担当者がFAQ候補登録
  ↓
管理者がFAQ候補を確認
FAQタイトル・回答を編集
  ↓
公開可否を判断
  ├─ 公開する → faq_items.is_public = 1
  └─ 公開しない → faq_items.is_public = 0
  ↓
依頼者がFAQ検索で閲覧
```

### 9.3 詳細手順

| Step | 実施者       | 操作                          | システム処理                           |
| ---: | ------------ | ----------------------------- | -------------------------------------- |
|    1 | 管理部担当者 | 完了済み問い合わせを確認する  | FAQ候補にできるか判断する              |
|    2 | 管理部担当者 | FAQ候補として登録する         | `inquiries.faq_candidate` を更新する   |
|    3 | 管理部担当者 | FAQタイトル・回答案を入力する | `faq_title`, `faq_answer` を保存する   |
|    4 | 管理者       | FAQ候補一覧を確認する         | FAQ候補を表示する                      |
|    5 | 管理者       | 公開FAQとして作成する         | `faq_items` に登録する                 |
|    6 | 管理者       | FAQ文面を編集する             | 個人情報や内部情報を除いた内容に整える |
|    7 | 管理者       | 公開状態にする                | `is_public = 1` に更新する             |
|    8 | システム     | 操作履歴を保存する            | `operation_logs` に保存する            |
|    9 | 依頼者       | FAQを検索・閲覧する           | `view_count` を加算する                |

### 9.4 FAQ公開時の確認項目

| 確認項目     | 内容                                               |
| ------------ | -------------------------------------------------- |
| 個人情報     | 個人名、メールアドレス、個別事情が含まれていないか |
| 内部情報     | 管理部内部の判断や担当者メモが含まれていないか     |
| 汎用性       | 他の依頼者にも役立つ内容か                         |
| 回答の明確性 | 問い合わせせずに理解できる文面か                   |
| カテゴリ     | 適切なカテゴリに分類されているか                   |

## 10. 通知対象確認フロー

### 10.1 目的

要対応アラートを、通知対象抽出と通知文生成に接続する。Ver.3では外部通知の実送信は行わず、通知対象確認と通知ログ保存までを対象とする。

### 10.2 基本フロー

```text
管理部担当者または管理者が通知対象確認画面を開く
  ↓
通知種別を選択
対象問い合わせを抽出
通知文を生成
通知ログに保存
  ↓
必要に応じて reviewed / skipped にする
```

### 10.3 通知種別別フロー

| 通知種別       | 対象                                 | 主な通知先     |
| -------------- | ------------------------------------ | -------------- |
| 期限前         | 期限が近い未完了問い合わせ           | 担当者         |
| 期限超過       | 期限を過ぎた未完了問い合わせ         | 担当者、管理者 |
| 担当者未設定   | 担当者が空欄の問い合わせ             | 管理者         |
| 情報待ち長期化 | 情報待ち状態が一定期間続く問い合わせ | 担当者         |
| ステータス変更 | ステータスが変更された問い合わせ     | 依頼者         |
| 完了           | 完了した問い合わせ                   | 依頼者         |

### 10.4 詳細手順

| Step | 実施者               | 操作                   | システム処理                                     |
| ---: | -------------------- | ---------------------- | ------------------------------------------------ |
|    1 | 管理部担当者・管理者 | 通知対象確認画面を開く | 通知種別選択欄を表示する                         |
|    2 | 管理部担当者・管理者 | 通知種別を選択する     | 条件に合う問い合わせを抽出する                   |
|    3 | 管理部担当者・管理者 | 対象一覧を確認する     | 問い合わせID、担当者、期限、ステータスを表示する |
|    4 | 管理部担当者・管理者 | 通知文を生成する       | 種別に応じた文面を作成する                       |
|    5 | 管理部担当者・管理者 | 通知ログに保存する     | `notification_logs` に保存する                   |
|    6 | 管理部担当者・管理者 | 不要な通知を除外する   | `status = skipped` にする                        |
|    7 | 管理部担当者・管理者 | 確認済みにする         | `status = reviewed` にする                       |

### 10.5 Ver.3での制約

| 制約               | 内容                                  |
| ------------------ | ------------------------------------- |
| 実送信なし         | メール・Slackへの送信は行わない       |
| 通知ログ保存まで   | 通知対象、通知文、通知状態を保存する  |
| 外部連携は後続課題 | 実送信はVer.4以降または将来課題とする |

## 11. 履歴管理フロー

### 11.1 目的

問い合わせ対応の経緯を追跡できるようにし、引き継ぎ、確認、運用改善に活用する。

### 11.2 操作履歴フロー

```text
主要操作実行
  ↓
操作内容を判定
  ↓
operation_logs に保存
  ↓
管理者が履歴確認画面で確認
```

### 11.3 操作履歴の登録対象

| 操作           | 登録内容                           |
| -------------- | ---------------------------------- |
| 問い合わせ作成 | 作成者、対象問い合わせID、作成日時 |
| 問い合わせ更新 | 更新者、対象問い合わせID、更新内容 |
| ステータス変更 | 変更者、変更前後のステータス       |
| 担当者変更     | 変更者、変更前後の担当者           |
| FAQ作成        | 作成者、FAQ ID                     |
| FAQ更新        | 更新者、FAQ ID、変更内容           |
| FAQ公開        | 公開者、FAQ ID                     |
| FAQ非公開      | 非公開にしたユーザー、FAQ ID       |
| コメント追加   | 投稿者、対象問い合わせID           |
| 通知文生成     | 実行者、通知ID、通知種別           |

### 11.4 ステータス履歴フロー

```text
問い合わせ更新画面でステータス変更
  ↓
inquiries.status を更新
  ↓
status_history に変更前後を保存
  ↓
問い合わせ詳細または履歴確認画面で確認
```

### 11.5 コメント履歴フロー

```text id="d1ywgu"
管理部担当者がコメント入力
  ↓
表示区分を選択
  ├─ internal → 管理部内部向け
  └─ requester → 依頼者にも表示可能
  ↓
inquiry_comments に保存
  ↓
問い合わせ詳細で時系列表示
```

### 11.6 履歴の表示権限

| 履歴               | requester | staff | admin | viewer |
| ------------------ | --------: | ----: | ----: | -----: |
| 操作履歴           |         - |     - |     ○ |      - |
| ステータス履歴     |         - |     △ |     ○ |      - |
| 内部コメント       |         - |     ○ |     ○ |      - |
| 依頼者向けコメント |         ○ |     ○ |     ○ |      - |

## 12. 集計・改善フロー

### 12.1 目的

Ver.3で追加したFAQ、通知、履歴情報を集計し、問い合わせ削減や対応漏れ防止の改善状況を把握する。

### 12.2 基本フロー

```text id="z2wffv"
管理者または閲覧者が集計画面を開く
  ↓
問い合わせ件数・対応状況を確認
FAQ活用状況を確認
通知対象件数を確認
履歴情報を確認
  ↓
Tableau用CSVを出力
  ↓
改善施策を検討
```

### 12.3 確認する指標

| 分類           | 指標                                           |
| -------------- | ---------------------------------------------- |
| 問い合わせ状況 | 総件数、未完了件数、完了件数、期限超過件数     |
| カテゴリ別状況 | カテゴリ別件数、カテゴリ別対応時間             |
| 担当者別状況   | 担当者別件数、担当者別アラート件数             |
| FAQ活用        | 公開FAQ件数、閲覧数、役立ち件数                |
| 通知対象       | 期限前、期限超過、担当者未設定、情報待ち長期化 |
| 履歴           | 操作件数、ステータス変更件数、コメント件数     |

### 12.4 改善判断

| 状況                           | 改善案                                     |
| ------------------------------ | ------------------------------------------ |
| 同じカテゴリの問い合わせが多い | FAQ追加、マニュアル整備                    |
| 期限超過が多い                 | 担当者配分、期限設定、通知条件の見直し     |
| 担当者未設定が多い             | 朝会・初期割当ルールの見直し               |
| 情報待ち長期化が多い           | 依頼者向け入力項目、確認テンプレートの改善 |
| FAQ閲覧数が多い                | FAQの品質改善、関連FAQ追加                 |
| FAQ閲覧が少ない                | FAQ検索導線、画面案内の改善                |

## 13. エラー・例外時の業務フロー

### 13.1 問い合わせ登録に失敗した場合

| 状況             | 対応                               |
| ---------------- | ---------------------------------- |
| 必須項目が未入力 | 画面上にエラーを表示し、登録しない |
| 希望期限が不正   | 日付入力を修正させる               |
| DB保存に失敗     | エラーを表示し、再実行を促す       |

### 13.2 FAQ公開に問題がある場合

| 状況                       | 対応                                                  |
| -------------------------- | ----------------------------------------------------- |
| FAQタイトルが空            | 公開不可とする                                        |
| FAQ回答が空                | 公開不可とする                                        |
| 個人情報を含む可能性がある | 管理者が編集してから公開する                          |
| 元問い合わせが削除・不明   | `source_request_id` なしのFAQとして扱うか、公開しない |

### 13.3 権限外操作が行われた場合

| 状況                           | 対応                         |
| ------------------------------ | ---------------------------- |
| requesterが管理画面へアクセス  | 利用不可メッセージを表示する |
| viewerが更新操作をしようとする | 更新ボタンを表示しない       |
| 未ログイン状態                 | ログイン画面を表示する       |

## 14. Ver.3業務フローの対象外

Ver.3では、以下の業務フローは対象外とする。

| 対象外                         | 理由                                                  |
| ------------------------------ | ----------------------------------------------------- |
| 本格的な承認ワークフロー       | 問い合わせ管理の主目的から外れ、仕様が大きくなるため  |
| 外部メール・Slack送信          | Ver.3では通知文生成と通知ログ保存までを対象とするため |
| 添付ファイル提出・確認         | ファイル管理、容量、権限設計が必要なため              |
| パスワード認証・権限マスタ管理 | 簡易ログインでVer.3の目的を満たせるため               |
| 多人数同時更新への厳密対応     | SQLite・Streamlit前提の小規模業務アプリとして扱うため |

## 18. 結論

Ver.3の業務フローでは、問い合わせ管理を管理部だけの作業から、依頼者・管理部担当者・管理者・閲覧者がそれぞれ関与する流れへ拡張する。

依頼者はFAQ検索によって自己解決を試み、解決しない場合のみ問い合わせを登録する。
管理部担当者はアラートを起点に対応すべき案件を確認し、更新・コメント・FAQ候補登録を行う。
管理者はFAQ公開、通知対象確認、履歴確認、集計確認を通じて、問い合わせ対応の品質と効率を管理する。
閲覧者は集計結果を確認し、組織全体の改善状況を把握する。

これにより、Ver.3では問い合わせ管理を「記録する業務」から、「自己解決、対応漏れ防止、履歴管理、改善分析を含む業務フロー」へ発展させる。
