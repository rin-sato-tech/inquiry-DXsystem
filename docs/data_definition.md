# データ定義書

## 1. 目的

本ドキュメントは、社内問い合わせ管理システムで使用するデータ項目を定義するものです。
主に、SQLiteのメインテーブルである `inquiries` のカラム、入力値の選択肢、派生項目、Tableau出力用の項目について整理します。

Ver.1では、問い合わせの受付、記録、進捗管理、期限超過判定、集計、Tableau連携を目的としたデータ構造を定義しました。
Ver.2では、以下の機能追加に伴い、データ項目を拡張しています。

- 要対応アラート
- FAQ候補管理
- カテゴリ別入力フォーム
- 依頼者向け確認画面
- Ver.2追加集計
- Tableau出力列の拡張

## 2. テーブル概要

SQLiteのメインテーブル名は `inquiries` とします。

| 項目         | 内容                                                                                                 |
| ------------ | ---------------------------------------------------------------------------------------------------- |
| テーブル名   | `inquiries`                                                                                          |
| 用途         | 社内問い合わせ・依頼情報の保存                                                                       |
| 主キー相当   | `request_id`                                                                                         |
| 保存先DB     | `data/inquiry.db`                                                                                    |
| 主な利用画面 | 問い合わせ一覧、新規登録、ステータス更新、要対応アラート、FAQ候補管理、依頼者向け確認、集計・CSV出力 |

## 5. 値の定義

### 5.1 department

部署は以下を想定します。

- 営業部
- 業務部
- 施工管理部
- 倉庫・物流部
- 管理部
- 経営層

### 5.2 channel

受付経路は以下を想定します。

- フォーム
- メール
- 電話
- 口頭
- チャット
- Excel

DX化後の新規登録は原則として「フォーム」を使用します。

### 5-3. category

問い合わせカテゴリは以下を想定します。

- PC・システム
- アカウント・権限
- 経費・請求
- 勤怠・労務
- 備品・設備
- 契約・書類
- 車両・配送
- その他

### 5-4. priority

優先度は以下を想定します。

- 高
- 中
- 低

### 5-5. status

ステータスは以下を想定します。

- 未対応
- 対応中
- 情報待ち
- 承認待ち
- 完了

## 6. ステータスの意味

| ステータス | 意味                                           |
| ---------- | ---------------------------------------------- |
| 未対応     | 受付済みだが、管理部がまだ対応を開始していない |
| 対応中     | 管理部が対応を進めている                       |
| 情報待ち   | 依頼者からの追加情報を待っている               |
| 承認待ち   | 上長や関係者の承認を待っている                 |
| 完了       | 対応が完了している                             |

## 7. `request_id` 採番ルール

問い合わせIDは以下の形式とします。

```text
REQ-YYYYMMDD-001
```

同日に複数登録される場合は、末尾の連番を増やします。

例：

```text
REQ-20260625-001
REQ-20260625-002
REQ-20260625-003
```

## 8. 日付・時刻の形式

| 項目     | 形式                | 例                    |
| -------- | ------------------- | --------------------- |
| 受付日   | `YYYY-MM-DD`        | `2026-07-01`          |
| 受付時刻 | `HH:MM`             | `09:30`               |
| 希望期限 | `YYYY-MM-DD`        | `2026-07-05`          |
| 完了日   | `YYYY-MM-DD`        | `2026-07-04`          |
| 作成日時 | ISO形式の日時文字列 | `2026-07-01T09:30:00` |
| 更新日時 | ISO形式の日時文字列 | `2026-07-01T10:15:00` |

日付比較や集計時には、Python側で `pandas.to_datetime()` を使って日付型に変換します。

## 9. `additional_info` の保存形式

`additional_info` は、カテゴリ別入力フォームで入力された値を複数行テキストとして保存します。

例：

```text
PC管理番号: PC-014
発生時刻: 09:30
エラー内容: 販売管理システム起動時に認証エラー
再起動有無: 実施済み
```

Ver.2では、カテゴリ別の追加情報を個別カラムや別テーブルには分けず、1つのテキストカラムに保存します。
これは、実装をシンプルに保ちつつ、管理部が確認すべき追加情報を一覧・詳細画面で確認できるようにするためです。

## 10. カテゴリ別追加項目

Ver.2では、カテゴリに応じて以下の追加項目を表示します。

| カテゴリ         | 追加項目                                                 |
| ---------------- | -------------------------------------------------------- |
| PC・システム     | PC管理番号、発生時刻、エラー内容、再起動有無             |
| アカウント・権限 | 対象システム、対象フォルダ・対象機能、必要な権限、承認者 |
| 勤怠・労務       | 対象日、修正前時刻、修正後時刻、理由                     |
| 経費・請求       | 金額、取引先、対象月、添付書類有無                       |
| 備品・設備       | 備品名・設備名、数量、利用目的、希望日                   |

上記の項目定義は `src/category_fields.py` で管理します。

## 11. 期限超過判定

期限超過は以下の条件で判定します。

```text
status != "完了" and due_date < 今日
```

この判定は、一覧表示、集計、Tableau出力で使用します。

## 12. Ver.2アラート判定

Ver.2では、期限超過に加えて以下のアラートを判定します。

| アラート種別   | 条件                                                 |
| -------------- | ---------------------------------------------------- |
| 期限超過       | 未完了、かつ希望期限が今日より前                     |
| 本日期限       | 未完了、かつ希望期限が今日                           |
| 期限間近       | 未完了、かつ希望期限が明日まで                       |
| 担当者未設定   | 担当者が空欄                                         |
| 情報待ち長期化 | ステータスが情報待ち、かつ受付日から一定日数以上経過 |

アラート判定は、DBには保存せず、Python側で派生列として計算します。

| `overdue_label`         | 期限超過かどうかの表示ラベル                 |

| `request_month`         | 受付日の年月                                 |
| `is_completed`          | ステータスが完了ならTrue                     |
| `is_completed_int`      | 完了なら1、それ以外は0                       |
| `is_open`               | ステータスが完了以外ならTrue                 |
| `is_open_int`           | 未完了なら1、それ以外は0                     |
| `management_hours`      | `management_minutes / 60`                    |
| `actual_response_hours` | `actual_response_minutes / 60`               |
| `total_work_hours`      | 管理作業時間と実対応時間の合計               |

### 13.2 Ver.2由来の派生項目 (14列)

| 項目                      | 内容                                       |

| `has_additional_info`     | `additional_info` が入力されていればTrue   |
| `has_additional_info_int` | 追加情報ありなら1、それ以外は0             |
| `is_requester_visible`    | 依頼者向け画面に表示するならTrue           |
| `requester_visible_int`   | 依頼者向け表示なら1、それ以外は0           |
| `requester_visible_label` | 依頼者向け表示・非表示のラベル             |

## 14. Tableau出力用の主な項目 (21列)

Tableau用CSVでは、DB保存項目に加えて、集計しやすい派生項目を出力します。

| 項目                      | 内容                                 |
| ------------------------- | ------------------------------------ |
| `request_id`              | 問い合わせID                         |
| `request_date`            | 受付日                               |
| `request_month`           | 受付月                               |
| `department`              | 部署                                 |
| `channel`                 | 受付経路                             |
| `category`                | カテゴリ                             |
| `priority`                | 優先度                               |
| `assignee`                | 担当者                               |
| `status`                  | ステータス                           |
| `status_group`            | ステータスの大分類                   |
| `overdue_int`             | 期限超過の0/1列                      |
| `response_days`           | 対応日数                             |
| `management_hours`        | 管理作業時間                         |
| `actual_response_hours`   | 実対応時間                           |
| `total_work_hours`        | 合計作業時間                         |
| `has_alert`               | 要対応アラート対象か                 |
| `alert_type`              | アラート種別                         |
| `faq_candidate_int`       | FAQ候補かどうか                      |
| `has_additional_info_int` | カテゴリ別追加情報が入力されているか |
| `requester_visible_int`   | 依頼者向け画面に表示するか           |
| `requester_visible_label` | 依頼者向け表示区分                   |

Tableauでは、0/1列を合計することで件数を集計できます。

例：

```text
SUM(faq_candidate_int) = FAQ候補件数
SUM(has_additional_info_int) = 追加情報あり件数
SUM(overdue_int) = 期限超過件数
```

また、0/1列の平均を取ることで割合を計算できます。

例：

```text
AVG(has_additional_info_int) = 追加情報入力率
```

## 15. データ設計上の方針

本プロジェクトでは、以下の方針でデータ設計を行います。

| 方針                         | 内容                                                               |
| ---------------------------- | ------------------------------------------------------------------ |
| 小規模運用を想定             | 従業員30名規模の社内問い合わせ管理を想定する                       |
| SQLiteを利用                 | 環境構築を簡単にし、ポートフォリオとして再現しやすくする           |
| 正規化しすぎない             | Ver.2では複雑なテーブル分割を避け、1テーブル中心で管理する         |
| 派生列はPython側で計算       | アラート、期限超過、入力率などはDB保存せず、表示・出力時に計算する |
| Tableau向けに0/1列を用意     | BIツールで件数・割合を集計しやすくする                             |
| 本格運用との差分を明確にする | ログイン認証、権限管理、FAQ公開ページなどは今後の課題とする        |

## 16. 今後のデータ設計上の課題

本格運用を想定する場合、以下のようなデータ設計の拡張が考えられます。

| 課題                   | 内容                                    |
| ---------------------- | --------------------------------------- |
| ユーザーテーブル       | 依頼者、管理部担当者、管理者を管理する  |
| 権限管理テーブル       | ロールごとの閲覧・更新権限を管理する    |
| FAQ専用テーブル        | 問い合わせ履歴から独立してFAQを管理する |
| FAQ編集履歴テーブル    | FAQ回答案の変更履歴を残す               |
| 操作履歴テーブル       | 誰がいつ問い合わせを更新したか記録する  |
| 添付ファイルテーブル   | スクリーンショットや証憑資料を管理する  |
| ステータス履歴テーブル | ステータス変更の履歴を管理する          |
| PostgreSQL移行         | 複数ユーザー・本格運用に対応する        |
