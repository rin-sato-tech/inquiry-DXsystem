# WBS6 学習ログ：依頼者向け確認画面

## 1. 依頼者向け確認画面の目的

依頼者向け確認画面の目的は、管理部への「今どうなっていますか」という確認問い合わせを減らすことである。問い合わせを登録した後、依頼者が状況を確認できない場合、以下のような追加問い合わせが発生しやすい。

- この依頼は受け付けられていますか
- 誰が担当していますか
- いつ頃対応されますか
- 対応は完了していますか
- 回答内容をもう一度確認したいです

依頼者が自分で状況を確認できれば、管理部が個別に進捗確認へ対応する負担を減らせる。
ただし、Ver.2では本格的なログイン認証や権限管理は実装せず、ポートフォリオ用のデモ画面として実装した。

## 2. `requester_visible` の役割

WBS2で追加した `requester_visible` は、依頼者向け画面に表示するかどうかを制御するためのフラグである。WBS6では、この値を使って、依頼者向け表示対象を制御した。

```text
requester_visible = 1
  → 依頼者向け画面に表示

requester_visible = 0
  → 依頼者向け画面には表示しない
```

Ver.2では初期値を1にしているため、通常の問い合わせは依頼者向け画面に表示される。
ただし、管理部内部の確認事項や依頼者に見せたくない内容が含まれる場合は、将来的に `requester_visible = 0` に設定することで非表示にできる。

## 3. `filter_requester_inquiries()` の役割

`filter_requester_inquiries()` は、依頼者向け確認画面の検索処理を担当する関数である。
`include_hidden=False` の場合は、`requester_visible=1` の問い合わせだけを対象にする。

処理の流れは以下である。

```text
DataFrame化
requester_visible を補正
非表示対象を除外 (include_hidden=False)
↓
問い合わせIDで検索
依頼者名で検索
結果を返す
```

問い合わせID検索・依頼者名検索では、部分一致を使っている。

```python
.str.contains(request_id, case=False, na=False)
```

部分一致にすることで、入力の手間を減らせる。ただし、本格運用では、同姓の社員が複数いる可能性や、他人の問い合わせが見えるリスクがある。そのため、Ver.3以降ではログイン認証を入れ、本人の問い合わせだけを表示する設計にする必要がある。

## 4. 詳細表示に `expander` を使う理由

検索結果は一覧表でも表示するが、問い合わせ内容や回答内容は長くなることがある。そのため、1件ごとの詳細は `st.expander()` で表示した。

```python
with st.expander(f"問い合わせID：{request_id}", expanded=len(display_df) == 1):
    ...
```

検索結果が1件だけの場合は、最初から展開するようにした。複数件ある場合は、必要な問い合わせだけ開いて確認できる。これにより、一覧性と詳細確認の両方を両立できる。

## 5. 本格運用との差分

Ver.2の依頼者向け画面は、簡易検索ベースである。一方、本格運用では以下が必要になる。

| 項目         | 本格運用で必要な理由                           |
| ------------ | ---------------------------------------------- |
| ログイン認証 | 本人確認が必要                                 |
| ユーザー管理 | 社員ごとの問い合わせを紐づける必要がある       |
| 権限管理     | 他人の問い合わせを見せないようにする必要がある |
| 監査ログ     | 誰がいつ閲覧・更新したかを残す必要がある       |
| 非表示制御   | 機密性の高い問い合わせを制限する必要がある     |

Ver.2では、これらを実装対象外とし、画面上にもその前提を明記した。

## 6. 動作確認の方法

WBS6では、以下の手順で動作確認を行う。

### 構文確認

```bash
python -m py_compile app.py src/requester_view.py
```

### requester_view 関数確認

```bash
python - << 'EOF'
from src.db import fetch_all_inquiries
from src.requester_view import (
    add_requester_view_columns,
    filter_requester_inquiries,
    get_requester_status_counts,
)

records = fetch_all_inquiries()
df = add_requester_view_columns(records)

print("全件:", len(df))
print("依頼者向け表示対象:", int(df["is_requester_visible"].sum()) if not df.empty else 0)
print()
print(get_requester_status_counts(df))
print()
print(filter_requester_inquiries(df, requester="吉田").head())
EOF
```

## 7. 今後の実装とのつながり

WBS6で追加した依頼者向け確認画面は、今後以下に展開できる。

| 今後の機能   | つながり                                   |
| ------------ | ------------------------------------------ |
| ログイン認証 | 本人の問い合わせだけを表示する             |
| 権限管理     | 依頼者、管理部、管理者で見える範囲を分ける |
| 通知機能     | ステータス変更時に依頼者へ通知する         |
| FAQ機能      | 依頼者が問い合わせ前にFAQを確認する        |
| 操作履歴     | 依頼者が問い合わせの対応経緯を確認する     |
| スマホ対応   | 依頼者が外出先から状況確認できる           |
| Tableau出力  | 依頼者向け表示対象件数を可視化する         |

WBS6では、管理部だけでなく依頼者側の利用体験も考慮した。
これにより、問い合わせ管理アプリは「管理部が処理するためのツール」から、「依頼者も状況を確認できる業務支援ツール」へ発展した。
