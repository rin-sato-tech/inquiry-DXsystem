# configディレクトリを作る意味

`config` ディレクトリを作る理由は、部署名・担当者名・カテゴリ名・ステータス名のような「変更されやすい選択肢」を、Pythonコードから切り離すためです。今回でいうと、以下のようなファイルです。

```text
config/
 ├─ departments.csv
 ├─ categories.csv
 ├─ assignees.csv
 ├─ status_master.csv
 ├─ channels.csv
 └─ priorities.csv
```

## 1. 何を管理しているのか

`config` は、システムの動きそのものではなく、業務上の選択肢・マスタ情報を管理します。

| ファイル            | 管理する内容   | 例                       |
| ------------------- | -------------- | ------------------------ |
| `departments.csv`   | 部署名         | 営業部、業務部、管理部   |
| `categories.csv`    | 問い合わせ分類 | PC・システム、経費・請求 |
| `assignees.csv`     | 管理部担当者   | 原田 健太、松尾 佳奈     |
| `status_master.csv` | ステータス     | 未対応、対応中、完了     |
| `channels.csv`      | 受付経路       | フォーム、メール、電話   |
| `priorities.csv`    | 優先度         | 高、中、低               |

Streamlitの画面では、これらを読み込んで selectbox や multiselect の選択肢に使います。

## 2. `config`を使わない場合

もし `config` を作らない場合、選択肢をPythonコードの中に直接書くことになります。

```python
departments = ["営業部", "業務部", "施工管理部", "倉庫・物流部", "管理部"]
categories = ["PC・システム", "アカウント・権限", "経費・請求"]
assignees = ["藤原 直子", "松尾 佳奈", "原田 健太", "石井 美咲"]
```

これは小さいうちは楽ですが、後で困ります。たとえば、担当者が変わったとします。

```text
原田 健太 → 山本 健太
```

このとき、Pythonコードを開いて修正する必要があります。さらに、同じリストを複数箇所に書いていたら、修正漏れが起きます。

## 3. configを使うメリット

<details>
<summary>メリット1：<strong>コードを触らずに選択肢を変更できる</strong></summary>

たとえば、担当者を追加したい場合、`assignees.csv` を編集するだけで済みます。

| assignee  |
| --------- |
| 藤原 直子 |
| 松尾 佳奈 |
| 原田 健太 |
| 石井 美咲 |
| 山本 健太 |

Pythonコードを修正しなくてよいので、開発者でなくても軽微な変更ができます。これは、原田君が異動・退職した場合の保守性にも関わります。

</details>

<details>
<summary>メリット2：<strong>属人化を減らせる</strong></summary>

このシステムでは、原田君が開発担当という設定でした。もし選択肢や業務ルールがすべてコードに埋め込まれていると、原田君しか修正できないシステムになります。
一方、部署・カテゴリ・担当者・ステータスを `config` に分けておけば、管理部の他の人でも最低限のメンテナンスができます。

| 変更内容         | configがない場合       | configがある場合           |
| ---------------- | ---------------------- | -------------------------- |
| 担当者を追加     | Pythonコード修正が必要 | `assignees.csv` を編集     |
| 部署名を変更     | Pythonコード修正が必要 | `departments.csv` を編集   |
| カテゴリを追加   | Pythonコード修正が必要 | `categories.csv` を編集    |
| ステータスを変更 | Pythonコード修正が必要 | `status_master.csv` を編集 |

</details>

<details>
<summary>メリット3：<strong>入力ミスを減らせる</strong></summary>

選択肢を自由入力にすると、表記ゆれが発生します。たとえば、部署名だけでもこうなります。

```text
営業部、営業、営業課、営業部門、営業チーム
```

この状態になると、集計が崩れます。本来は同じ部署なのに、別カテゴリとして集計されます。`config` から選択させれば、表記を統一できます。これはTableauで集計するときにも重要です。

</details>

<details>
<summary>メリット4：<strong>Streamlit画面とデータ定義を一致させやすい</strong></summary>

`config` を使うと、Streamlitの登録画面・更新画面・一覧フィルタで同じ選択肢を使えます。

```text
新規登録画面
ステータス更新画面
一覧フィルタ
集計処理
```

これらが別々の選択肢を持つと、不整合が起きます。たとえば、新規登録では「情報待ち」が選べるのに、一覧フィルタでは「情報待ち」が出ない、という状態です。`config` を一元化しておくと、このズレを防げます。

</details>

<details>
<summary>メリット5：<strong>ポートフォリオとして設計力を示せる</strong></summary>

`config` を分けていると、単に動くアプリではなく、保守性を考えた業務アプリに見えます。

面接などでは、次のように説明できます。

> 担当者名、部署名、カテゴリ名、ステータス名は業務変更に伴って変わる可能性があるため、Pythonコードに直接書かず、config配下のCSVとして分離しました。これにより、軽微なマスタ変更であればコード修正なしで対応でき、開発担当者への属人化を抑えられるようにしました。

これはかなり実務的な説明です。

</details>

## 4. configに向いているもの・向いていないもの

すべてを `config` にすればよいわけではありません。

### configに向いているもの

| 種類       | 理由                           |
| ---------- | ------------------------------ |
| 部署名     | 組織変更で変わる               |
| 担当者名   | 異動・退職・追加がある         |
| カテゴリ名 | 運用しながら見直す可能性がある |
| ステータス | 業務フローに依存する           |
| 優先度     | 選択肢として固定したい         |
| 受付経路   | 集計軸として統一したい         |

### configに向いていないもの

| 種類           | 理由                             |
| -------------- | -------------------------------- |
| DB接続処理     | プログラムロジックそのもの       |
| 期限超過判定   | 業務ルールではあるが処理ロジック |
| 集計処理       | Pythonで実装する処理             |
| 画面レイアウト | Streamlit側で制御する内容        |

つまり、`config` は選択肢やマスタ情報に向いています。処理そのものは `src/` に置くのが自然です。

## 5. 今回のシステムでの位置づけ

今回の構成では、役割はこうです。

```text
config/
業務上の選択肢を管理する

src/
処理ロジックを書く

data/
問い合わせデータやSQLite DBを置く

app.py
Streamlit画面を作る
```

具体的には、`app.py` が `src/master_data.py` を通じて `config` を読み込みます。

```text
config/*.csv
↓
src/master_data.py
↓
app.py
↓
Streamlitのselectbox / multiselect
```

この構造にすることで、画面上の選択肢がCSVから決まります。

## 6. デメリットもある

もちろん、`config` を分けることにも少しデメリットはあります。

| デメリット                   | 内容                                               |
| ---------------------------- | -------------------------------------------------- |
| ファイル数が増える           | 小規模アプリでは少し大げさに見える                 |
| CSVの列名ミスに弱い          | `assignee` などの列名を間違えると読み込めない      |
| 管理対象が増える             | `config`ファイル自体を正しく管理する必要がある     |
| 動的反映には再読み込みが必要 | 変更後にアプリ再起動や再読み込みが必要な場合がある |

ただし、今回のようなポートフォリオ兼小規模業務アプリでは、メリットの方が大きいです。

## 7. 最低限どの`config`が必要か

最初から全部なくても動きます。ただ、今回のシステムでは以下は作っておく価値があります。

| 優先度 | ファイル            | 理由                   |
| ------ | ------------------- | ---------------------- |
| 高     | `departments.csv`   | 部署別集計に使う       |
| 高     | `categories.csv`    | 問い合わせ分類の中核   |
| 高     | `assignees.csv`     | 担当者別負荷を見る     |
| 高     | `status_master.csv` | 進捗管理に必須         |
| 中     | `channels.csv`      | DX化前の分散経路を示す |
| 中     | `priorities.csv`    | 優先度管理に使う       |

## 結論

`config` ディレクトリを作る理由は、部署名・カテゴリ名・担当者名・ステータス名のような変更されやすい情報を、Pythonコードから分離するためです。
メリットは、保守性が上がる、表記ゆれを防げる、コード修正なしで軽微な変更ができる、原田君への属人化を減らせる、画面と集計の選択肢を統一できることです。今回のような業務効率化システムでは、`config` は単なる補助ディレクトリではなく、業務ルールやマスタ情報を管理するための仕組みとして意味があります。
